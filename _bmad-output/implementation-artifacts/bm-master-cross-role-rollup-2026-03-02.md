# BM Master 跨角色推进清单（2026-03-02）

## 1. 目的

将当前产品与架构对齐后的关键推进事项，拆分为可由 BM Master 统一编排的跨角色执行清单，确保“门禁真相源一致、PG-only 落地、运行态可观测闭环”在同一迭代内收口。

## 2. 本轮总目标（统一口径）

1. 发布门禁、看板展示、测试结果三者同源且可对账。
2. 主链路彻底 PG-only，不允许 SQLite 回流。
3. Runtime Observability V2 达到可运营下钻能力（任务、风险、版本、工作包链路）。

## 3. 统一推进顺序（建议）

1. `PGO-S2/PGO-S4`：先修 PG-only 验收与主线测试口径。
2. `S2-C3/ARC-S7`：再统一门禁与看板真相源。
3. `S2-1/S2-14`：补齐运行态可观测 API 与链路下钻。
4. `PGO-S5`：最后加固 CI 回流硬门，防止反复。

## 4. 角色分工与事项

### 4.1 BM Master（总控编排）

- `P0` 建立单一发布结论口径（GO/NO_GO 仅一处输出），禁止多源冲突。
- `P0` 组织每日 1 次 cross-role 对账：测试结果、看板摘要、发布判定。
- `P1` 维护依赖顺序与阻塞台账（阻塞必须带责任人和解除条件）。

验收：
- 产出统一推进看板与阻塞清单；
- 当日所有 `P0` 条目有明确状态（进行中/阻塞/完成）。

### 4.2 BMM-PM（需求与节奏）

- `P0` 将以下事项创建为 Linear Issues 并关联 PR：
  - `P0-1` 门禁真相源统一（S2-C3 + ARC-S7）
  - `P0-2` PG-only 验收链路修复（PGO-S2 + PGO-S4）
  - `P0-3` No-Fallback 失败语义统一（ARC-S5）
  - `P0-4` 运行态四类核心 API 口径固定（S2-1）
  - `P1-1` 新增治理包链路观测（S2-14）
  - `P1-2` SQLite 回流硬门（PGO-S5）
- `P1` 每个 Issue 附 DoD、测试入口、证据路径模板。

验收：
- Linear 中全部事项可追踪，且每个事项都有验收标准与证据字段。

### 4.3 BMM-Architect（架构收口）

- `P0` 输出“观测数据真相源边界说明”：Dashboard/API 仅消费 PG 聚合结果。
- `P0` 审核 `Router -> Service -> Repository` 分层，不允许 Router 直拼 SQL。
- `P1` 审核 No-Fallback 与 PG-only 约束是否在代码路径完全生效。

验收：
- 架构检查结论文档更新；
- 对应 PR 获得架构评审通过结论。

### 4.4 BMM-Dev（实现）

- `P0` 修复 PG-only 验收脚本与主线测试环境依赖：
  - 缺失 `DATABASE_URL=postgresql://...` 必须 fail-fast，错误语义清晰。
- `P0` 修复看板构建逻辑：
  - 门禁状态、发布结论、回归状态一致；
  - recent_changes 按时间倒序；
  - blocked 判定不误伤“无 blocker”语义。
- `P1` 落地 S2-14 所需聚合接口与事件字段映射。

验收：
- 单测/集成测试通过；
- 看板与门禁产物同批次对账通过；
- API 响应结构满足运行态页面消费契约。

### 4.5 BMM-QA（测试与发布质量）

- `P0` 建立“门禁一致性回归套件”：
  - 用例 1：release_gate 失败时，dashboard `overall` 不得为 true。
  - 用例 2：回归失败时，发布结论必须 NO_GO。
  - 用例 3：PG-only 环境缺失配置时，验收脚本应失败且提示明确。
- `P0` 对 P0 事项执行发布前回归并给出 Go/No-Go 建议。
- `P1` 补充 S2-14 端到端链路观测回归（CLI -> Agent -> LLM -> Runtime）。

验收：
- 形成回归报告（含失败分类、影响范围、阻断建议）；
- 未通过用例必须阻断发布。

### 4.6 BMM-SM（执行监督）

- `P0` 跟踪跨角色依赖，保证顺序执行不打架（先 PG-only、再真相源、后观测增强）。
- `P1` 对延期事项触发风险升级与节奏调整建议。

验收：
- 每日节奏报告可复盘，阻塞清单不过夜（除外部依赖）。

## 5. 交付物清单（统一归档）

1. 代码与测试 PR（关联 Linear Issue）。
2. 回归报告（`output/workpackages/*` 或 `output/test-reports/*`）。
3. 看板快照（`output/dashboard/*`）与门禁报告对账记录。
4. 发布结论记录（GO/NO_GO）及原因说明。

## 6. 发布门槛（本轮）

满足以下条件才可建议 `GO`：

1. P0 事项全部完成且无未豁免阻塞。
2. PG-only 主线测试通过，且无 SQLite 回流命中。
3. 门禁报告、看板摘要、发布结论三者一致。
4. QA 回归报告结论为可发布。
