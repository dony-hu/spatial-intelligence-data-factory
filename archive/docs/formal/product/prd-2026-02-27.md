# PRD - 空间智能数据工厂（Level 4）

## 1. 文档信息

- 文档版本：v1.0
- 创建日期：2026-02-27
- 项目名称：spatial-intelligence-data-factory
- 项目类型：other（平台工程）
- 项目级别：Level 4
- 文档语言：中文（遵循仓库 AGENTS 规则）

## 2. 背景与问题

当前平台已具备地址治理、治理 API、工作包执行、可观测看板与 CI 门禁基础能力，但仍存在以下核心问题：

1. 多业务线共享能力与地区定制边界仍需持续收敛，避免平台能力被定制逻辑侵蚀。
2. 质量门禁、运行态证据与交付态验收标准尚未形成统一“可执行契约”。
3. 执行链路存在多入口与多运行模式并存，长期维护复杂度较高。
4. 文档、脚本、服务与工作包产物增长较快，工程治理成本上升。

## 3. 产品愿景与目标

建设“可复用、可治理、可审计、可持续演进”的空间智能数据工厂平台，支撑多地区、多场景稳定交付。

阶段目标（本 PRD 对应）：

1. 固化地址治理与可信数据融合的核心交付主链路。
2. 建立从需求到上线的统一质量门禁和证据回传机制。
3. 将关键能力沉淀为可复用工作包与标准 API 能力。
4. 形成 Build-time（研发）与 Run-time（运营）角色边界清晰的执行模型。

## 4. 范围定义

### 4.1 In Scope

1. 地址治理主链路：标准化、解析、召回、评分、人工复核闭环。
2. 治理 API：任务提交、结果查询、规则集管理、运维汇总、Lab 能力。
3. 执行层：队列分发（sync/rq/in_memory）、任务状态机与结果落库。
4. 可信数据 Hub：多源可信数据接入、查询与校验接口。
5. 质量门禁：P0 工作包门禁、夜间质量门禁、SQL 只读安全门禁、Web E2E。
6. 可观测交付：运行指标、测试看板、工作包报告产物。

### 4.2 Out of Scope

1. 地区专属流程与一次性定制逻辑（保留在子项目层）。
2. 面向全公司通用低代码平台能力建设。
3. 本期以外的大规模历史数据全量迁移。

## 5. 关键用户与场景

1. 平台研发：实现能力沉淀、模块拆分、跨环境可运行交付。
2. 数据治理与质检：配置规则、查看质量结果、执行人工复核。
3. 运维与项目管理：读取门禁状态、追踪风险、管理发布窗口。
4. 业务使用方：消费稳定 API 与可信结果数据。

核心场景：

1. 地址记录批量治理并返回可解释证据。
2. 低置信度结果进入人工复核与审核闭环。
3. 工作包执行时产生可追踪报告并受 CI 门禁约束。
4. 可信数据多源验证与查询服务统一输出。

## 6. 功能需求（按 Epic）

### EPIC A：地址治理与结果可信化

目标：确保地址治理主链路稳定、可解释、可复核。

Story 示例：

1. A1 任务提交后可在统一状态机中从 `PENDING` 到 `SUCCEEDED/FAILED` 演进。
2. A2 每条结果必须返回 `strategy/confidence/evidence` 字段。
3. A3 低置信度或冲突结果可进入人工审核队列并支持决策回写。

验收标准：

1. 核心 API 契约稳定，回归测试通过。
2. 结果记录可关联 task_id、raw_id、ruleset 版本。
3. 失败路径有明确错误语义，不出现“无产物但成功”。

### EPIC B：治理控制面与执行编排

目标：让执行模式、运行时选择与任务落库具备可控性和可观测性。

Story 示例：

1. B1 队列模式支持同步、RQ、内存回退，且可通过环境变量控制。
2. B2 运行时选择策略明确，非法模式可快速失败。
3. B3 任务执行过程可写入关键审计字段与结果摘要。

验收标准：

1. 队列与运行时策略有自动化测试覆盖。
2. 在缺少外部依赖时可本地完成最小链路验证。
3. 对降级路径（fallback）有明确开关与风险说明。

### EPIC C：质量门禁与可观测交付

目标：将测试、门禁、报告、看板连成闭环，支持发布 Go/No-Go 决策。

Story 示例：

1. C1 P0 工作包门禁可阻断不符合哈希/回归约束的提交。
2. C2 夜间质量门禁可执行重试、失败分类与告警产物输出。
3. C3 Web E2E 结果可汇总进测试状态看板。

验收标准：

1. CI 工作流可稳定产出门禁结果与证据文件。
2. 门禁失败时能定位到具体套件与失败原因。
3. 看板与报告数据来源一致，支持复盘追踪。

### EPIC D：工程治理与平台可维护性

目标：控制仓库复杂度，提升跨环境可复现性与可维护性。

Story 示例：

1. D1 统一 Python 环境与依赖管理，消除不可移植环境依赖。
2. D2 逐步拆分超大模块，降低变更冲击面。
3. D3 规范产物目录与归档策略，减少仓库噪音。

验收标准：

1. 新环境可按文档完成安装并运行最小测试集。
2. 大模块拆分后行为不变，回归测试通过。
3. 工程规范文档与自动化检查保持一致。

## 7. 非功能需求

1. 可用性：关键 API 与门禁脚本需可重复执行并给出稳定返回。
2. 可观测性：关键链路必须记录任务状态、失败原因、证据链接。
3. 可审计性：关键决策与发布动作需有日志与追踪信息。
4. 安全性：不在仓库提交明文密钥，遵循最小权限原则。
5. 可移植性：开发环境应可在新机器快速重建。

## 8. KPI 与成功标准

1. 关键门禁（P0 + Nightly）稳定通过率持续提升。
2. 地址治理主链路失败自动识别率与可解释性满足验收。
3. 关键问题从发现到定位时间持续下降。
4. 平台能力复用率持续提升（跨地区复用）。

## 9. 依赖与风险

关键依赖：

1. PostgreSQL/Redis 等基础依赖可用性。
2. 多源可信数据提供方稳定性与调用约束。
3. 测试数据与契约文件持续维护。

主要风险：

1. 降级路径长期化导致线上行为与预期偏差。
2. 多入口并存导致维护与排障成本增加。
3. 产物与环境文件入库造成仓库体积与可移植性问题。

## 10. 里程碑

1. M1（当前）：完成 PRD/架构/工作包基线对齐。
2. M2：治理主链路与门禁链路稳定化（含回归与证据闭环）。
3. M3：工程治理专项（环境、模块拆分、目录治理）。
4. M4：多地区能力复用与运行指标持续优化。

## 11. 发布与验收门禁

发布前必须满足：

1. 关键自动化测试通过（API、核心工作包、必要 E2E）。
2. 质量门禁产物完整（JSON/Markdown 报告与告警记录）。
3. 关键风险具备回滚方案与责任人。
4. 文档与日志更新完整，证据路径可追踪。
