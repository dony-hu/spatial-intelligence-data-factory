# BMAD 多项目组交付手册（面向非软件工程背景团队）

## 1. 目标

为上海、昆山、吴江、北京、深圳宝安等项目组提供统一的 BMAD 执行方法，确保：

- 每个阶段都有明确产出，不再停留在“口头推进”。
- 项目型交付习惯逐步过渡到“可复用能力交付”。
- 跨地域协作时，需求、决策、风险都可追踪。

## 2. 适用对象与角色

- 项目负责人（业务负责人）
- 需求负责人（PM/产品）
- 技术负责人（架构/开发）
- 交付负责人（实施/运维）
- 数据治理与质检人员

## 3. BMAD 阶段与必交付物

## 3.1 Phase 1 - Analysis（问题定义）

目标：把“要做什么、为什么做、做到什么程度”说清楚。

必交付物：

1. `product-brief.md`
2. 场景清单（含优先级）
3. 范围边界（In Scope / Out of Scope）
4. 北极星指标与阶段 KPI

完成门禁（全部满足才可进入 Planning）：

- 业务目标可量化。
- 范围边界已评审通过。
- 前两优先场景已确定。
- 风险与约束已记录。

## 3.2 Phase 2 - Planning（需求规划）

目标：把目标拆成可执行、可验收、可排期的需求结构。

必交付物：

1. `PRD`（按能力域拆解，不按地区拆解）
2. Epic -> Story -> Task 清单
3. 验收标准（每个 Story 必须有）
4. 发布范围与回滚原则

完成门禁：

- 每个需求都有负责人和验收标准。
- 关键依赖与阻塞路径已识别。
- 迭代优先级达成一致。

## 3.3 Phase 3 - Solutioning（方案设计）

目标：建立跨项目可复用、可演进的技术方案。

必交付物：

1. `architecture.md`
2. 数据契约（字段、质量规则、版本策略）
3. 服务契约（API、SLA、鉴权、限流）
4. 迁移计划与回滚方案

完成门禁：

- 架构评审通过。
- 兼容性与迁移策略明确。
- 变更影响范围可追踪。

## 3.4 Phase 4 - Implementation（实施交付）

目标：按迭代稳定交付并形成可审计证据。

必交付物：

1. Sprint 计划与燃尽跟踪
2. Story 实施记录与代码评审记录
3. 测试证据（功能、回归、关键路径）
4. 发布说明、监控指标与复盘记录

完成门禁：

- 合并前质量门禁全部通过（lint/test/build）。
- 发布有回滚方案和责任人。
- 日志与证据链接齐全。

## 4. 团队执行节奏（固定化）

1. 每日 15 分钟站会：仅同步进展、阻塞、当日目标。
2. 每周 1 次阶段评审：基于 `/workflow-status` 判断是否进阶段。
3. 每两周 1 次方案评审：仅讨论契约与变更影响。
4. 每月 1 次复盘：复用率、质量趋势、跨团队协作问题。

## 5. 新手友好的执行规则（必须遵守）

1. 先文档后开发：没有验收标准不得开工。
2. 先评审后变更：跨团队影响需求必须过评审。
3. 先复用后定制：能复用平台能力就不得重复造轮子。
4. 先记录后结束：当天必须补齐日志与证据链接。

## 6. 跨地域协同沟通模板（最小必填）

- 需求背景：
- 业务价值：
- 影响地区：
- 是否共性能力：
- 验收标准：
- 风险：
- 计划上线时间：
- 责任人：

## 7. 绩效与治理指标（建议）

1. 平台能力复用率（北极星）
2. 需求准时交付率
3. 缺陷逃逸率（上线后问题）
4. 需求返工率（因范围不清导致）
5. 跨团队阻塞时长

## 8. 落地顺序（30 天）

1. 第 1 周：所有项目组完成 Analysis 工件补齐。
2. 第 2 周：完成统一 PRD 模板与 Story 拆分。
3. 第 3 周：完成共性能力架构评审。
4. 第 4 周：按统一门禁进入实施并进行首轮复盘。
