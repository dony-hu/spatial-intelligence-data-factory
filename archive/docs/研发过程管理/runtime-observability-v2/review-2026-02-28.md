# 评审纪要：Runtime Observability V2 Epic 集体评审（2026-02-28）

## 1. 评审范围

1. PRD：`docs/epics/runtime-observability-v2/product.md`
2. 架构：`docs/epics/runtime-observability-v2/design.md`
3. Epic：`docs/epics/runtime-observability-v2/epic.md`
4. Story：`OBS-RUNTIME-S2-1` ~ `OBS-RUNTIME-S2-4`

## 2. 评审角色

1. PM：目标、范围、验收完整性
2. Architect：边界、依赖、耦合风险
3. Dev Lead：可实施性、测试闭环、实现顺序
4. QA：可验证性、回归策略、证据要求

## 3. 评审结论

结论：**有条件通过（Go）**，按 Story 顺序执行。

通过条件：

1. 严格使用 PG 运行库作为唯一真相源。
2. 关键链路保持 No-Fallback，不允许 in-memory/Dummy runtime。
3. S2-1 先完成并稳定后，再推进 S2-2/S2-3。

## 4. 关键发现

1. 现状问题
   - 页面信息层级偏研发过程，不利于运营值班决策。
   - 无样例数据时页面价值接近空白。

2. 方案有效点
   - 运行态 API 与页面分层清晰，指标口径可统一。
   - 样例包机制能快速建立可观测演示与回归基线。
   - 下钻追溯可支撑问题定位与复盘闭环。

3. 主要风险
   - 指标口径若分散在前端会产生不一致。
   - 灌入样例分布不稳会导致验收波动。
   - 审计事件缺字段会影响下钻完整性。

## 5. 决议与行动项

1. 决议 D1
   - 所有核心 KPI 仅由后端聚合服务输出。
2. 决议 D2
   - 样例包采用固定分层配比与固定 seed，确保可复现。
3. 决议 D3
   - 任务详情必须包含 evidence 与 review 字段，不允许“空壳下钻”。
4. 行动 A1（Dev）
   - 先完成 S2-1 API 契约与口径测试，再上实现。
5. 行动 A2（Dev + QA）
   - 完成 S2-3 样例灌入回归测试与指标区间校验。
6. 行动 A3（PM + QA）
   - 对照 PRD 验收条款出具验收报告模板。

## 6. 进入开发门禁

以下三项满足后进入全量开发：

1. S2-1 的失败测试样例已补齐并可执行。
2. 运行态 API 草契约冻结（字段名与错误语义冻结）。
3. 样例包最小字段清单冻结（task/result/review/evidence）。
