# Story: S1-A1 治理任务状态机稳定化

## 目标

保证治理任务状态在主链路中稳定按 `PENDING -> RUNNING -> SUCCEEDED/FAILED` 演进，避免异常分支出现状态错乱。

## 验收标准

1. 任务提交后初始状态为 `PENDING`。
2. 执行开始后状态更新为 `RUNNING`。
3. 正常执行完成后状态为 `SUCCEEDED`，失败时状态为 `FAILED`。
4. 任一异常路径不允许返回空状态或未定义状态。

## 开发任务

1. 先补测试：增加状态流转失败用例（提交失败、执行异常、队列降级）。
2. 再改实现：校正 `tasks` 路由与 `governance_job` 状态写入时机。
3. 最后验证：运行目标测试集并输出状态流转证据。

## 测试用例

1. 正常提交+执行：最终 `SUCCEEDED`。
2. 模拟 worker 异常：最终 `FAILED`。
3. 队列不可用降级：状态仍可闭环到终态。

## 对齐信息（PRD/架构）

1. PRD 对齐：EPIC A-A1（状态机）+ EPIC B-B1/B2（运行模式可控）。
2. 架构对齐：
- `docs/architecture/system_overview.md` 治理主链路状态机要求。
- `docs/architecture/module_boundaries.md` API/Worker/Core 边界。

## 模块边界与 API 边界

1. 所属模块：`governance_api`、`runtime worker`、`governance repository`。
2. 上游入口：任务提交与调度触发。
3. 下游依赖：状态持久化、审计写入。
4. API 边界：状态查询从数据库读取，不读取进程内临时状态作为真相源。

## 依赖与禁止耦合

1. 允许依赖：`api/worker -> repository`。
2. 禁止耦合：
- 状态机逻辑散落在脚本侧并绕过统一服务层。
- 非定义状态写入数据库或返回体。
