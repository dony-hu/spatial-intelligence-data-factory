# Story: S1-A3 人工审核闭环补齐

## 目标

让低置信度结果进入人工审核队列并完成决策回写，确保审核前后状态、结果与审计信息一致可追踪。

## 验收标准

1. 待审核任务可被查询到且包含必要上下文字段。
2. 审核通过/拒绝后，结果状态与审核结论一致。
3. 审核动作具备审计信息（审核人、时间、决策）。
4. 审核接口异常时返回明确错误语义。

## 开发任务

1. 先补测试：增加审核队列查询、通过、拒绝、异常分支用例。
2. 再改实现：完善 `manual_review` 路由与仓储回写逻辑。
3. 最后验证：执行审核相关测试并检查审计字段落地。

## 测试用例

1. 待审核记录可查询且计数正确。
2. 审核通过后状态更新为已决策。
3. 审核拒绝后保留原因并可追溯审核人。

## 对齐信息（PRD/架构）

1. PRD 对齐：EPIC A-A3（人工复核闭环）+ EPIC C（审计可追踪）。
2. 架构对齐：
- `docs/architecture/system_overview.md` 治理主链路与审计要求。
- `docs/architecture/dependency_map.md` `audit` schema 依赖。

## 模块边界与 API 边界

1. 所属模块：`manual_review service`、`governance repository`、`audit log`。
2. 上游入口：审核列表查询、审核决策提交。
3. 下游依赖：审核状态回写、审计事件记录、结果重算（如需要）。
4. API 边界：审核 API 只处理审核语义，不直接操作任务执行编排。

## 依赖与禁止耦合

1. 允许依赖：`review api -> review service -> repository/audit`。
2. 禁止耦合：
- 在前端页面直接写库完成审核决策。
- 审核回写不记录审核人/时间/结论。
