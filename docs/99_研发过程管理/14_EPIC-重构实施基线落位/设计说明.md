# 设计说明

## 1. 设计原则

本主题遵循以下五条原则：

1. 先稳契约，再动结构。
2. 先冻结 legacy 与 target 边界，再做目录和模块迁移。
3. 拆分优先于替换，保持外部正式契约稳定。
4. 测试必须绑定现行正式边界，而不是继续追随历史实现。
5. 并行开发以共享契约为上游，以 baseline 回归为门禁。

## 2. 本主题的结构化拆解

### 2.1 契约层

本主题先收敛以下共享契约：

1. `workpackage_schema.v1`
2. `orchestration_context.v1`
3. Agent 与 Runtime 交接载荷
4. 数据库分域与迁移 owner

要求如下：

1. 共享契约必须先形成唯一消费口径。
2. 共享契约未冻结前，下游 Lane 不得自行推导接口。
3. 共享契约类改动默认由上游 owner 发独立 PR。

### 2.2 代码层

本主题的代码改造重点不是重写所有逻辑，而是建立“可持续重构”的目标结构：

1. `packages/factory_agent/`
   - 先拆 façade、conversation、blueprint、trace、memory、publish / dryrun 协调逻辑。
2. `services/governance_api/`
   - 先按领域拆 service，再拆分大 router。
3. `services/governance_worker/`
   - 保住现行 `workpackage executor` 主链，不继续兼容错误的旧 pipeline 假设。
4. `src/`
   - 视为 legacy/compat 过渡面，不再承接新增正式能力。

### 2.3 测试层

本主题的测试策略如下：

1. 先建立 smoke baseline。
2. 再建立 known-fail 清单。
3. 对仍绑定旧链路的测试，优先修正 seam，再继续功能改造。
4. 所有结构拆分都应以“外部契约不变”为前提补充或更新测试。

### 2.4 集成层

本主题沿用并行开发主题的集成规则，并补充以下执行策略：

1. 每日只允许进入一个跨 Lane 共享契约类 PR。
2. 集成值班位负责确认“今天合哪一个 PR”。
3. 单个 PR 不同时承接：
   - 契约变更
   - 目录迁移
   - 大规模实现迁移
4. 每个 Lane 合并前必须声明：
   - owned surface
   - 影响的共享契约
   - 最小测试集
   - 下游通知对象

## 3. 执行顺序设计

### 3.1 首轮执行顺序

本主题首轮执行顺序固定如下：

1. `EPIC4-S1`：共享契约冻结与消费基线固化
2. `EPIC4-S2`：legacy 边界冻结与目录归属声明
3. `EPIC4-S3`：Factory Agent 巨石拆分但保持外部契约不变
4. `EPIC4-S4`：Runtime / API seam 收敛与主链边界清理
5. `EPIC4-S5`：旧测试口径清理与 baseline 建立
6. `EPIC4-S6`：集成回归与首轮并行开发准入

### 3.2 并行与串行关系

1. `S1` 是所有实现 Story 的上游。
2. `S2` 是目录归属和模块迁移的前置约束。
3. `S3` 与 `S4` 可并行，但都依赖 `S1 / S2`。
4. `S5` 必须跟随 `S3 / S4` 同步推进，不能在末尾补。
5. `S6` 负责统一收口，不承接新的结构性改造。

## 4. 过渡态治理要求

当前仓库已经出现新旧实现并存，因此本主题补充如下要求：

1. 新能力默认落到目标 owned surface，不允许继续在旧壳中“顺手补一下”。
2. 若旧目录仍需保留，仅作为兼容层，不再承担新增正式能力。
3. 目录迁移、测试迁移与契约迁移必须拆开执行。
4. 任何会改变正式边界的改造，必须同步回填对应 Ring1 正式文档。

## 5. 本主题与后续主题的关系

本主题不是最终重构终点，而是后续替换动作的前置条件。

本主题完成后，才适合启动：

1. Agent 框架替换试点
   - 例如 `LangGraph + PydanticAI`
2. Runtime 控制层替换试点
   - 例如 `Go + Temporal`
3. 内部服务契约升级试点
   - 例如 `Buf + Protobuf + ConnectRPC`
