# Story: S1-A2 结果契约强校验

## 目标

确保治理结果契约字段完整且结构一致，关键字段缺失时返回明确错误语义，不进入“看似成功”的不完整结果。

## 验收标准

1. 结果对象必须包含 `strategy`、`confidence`、`evidence`。
2. `confidence` 必须可解析且在合法范围内。
3. 缺失关键字段时 API 返回可识别错误信息。
4. 合法结果可被下游查询接口稳定消费。

## 开发任务

1. 先补测试：增加关键字段缺失/类型不匹配失败用例。
2. 再改实现：补齐模型层和结果组装层的校验逻辑。
3. 最后验证：执行契约相关测试并校验错误返回体。

## 测试用例

1. 缺失 `strategy`：返回契约错误。
2. `confidence` 非数值：返回契约错误。
3. 完整合法结果：任务结果查询成功且字段齐全。

## 对齐信息（PRD/架构）

1. PRD 对齐：EPIC A-A2（结果契约）+ EPIC C（门禁证据一致）。
2. 架构对齐：
- `docs/02_总体架构/系统总览.md` 中 Core 结果契约要求。
- `docs/02_总体架构/模块边界.md` API 与 Core 边界。

## 模块边界与 API 边界

1. 所属模块：`address_core`、`application service`、`governance_api`。
2. 上游入口：治理执行结果写回。
3. 下游依赖：契约校验器、仓储写入、查询接口。
4. API 边界：结果契约错误必须显式返回，不允许自动补字段“纠正成功”。

## 依赖与禁止耦合

1. 允许依赖：`core -> contract validator -> service/api`。
2. 禁止耦合：
- 在路由层临时改写核心结果字段掩盖上游错误。
- 跳过契约校验直接持久化不完整结果。
