# 架构设计：系统运行态可观测（Runtime Observability V2）

## 1. 设计目标

围绕“地址治理系统运行成果可观测”构建 V2 架构，确保页面展示的是业务运行状态，而非研发过程状态。

目标：

1. 可直接观测治理产出：任务量、完成率、阻塞率、置信度、审核积压
2. 可快速定位风险：阻塞原因 Top、低置信模式 Top、版本对比
3. 可下钻追溯证据：任务 -> 结果 -> evidence -> 审核记录 -> 审计事件
4. 严格 No-Fallback：关键数据源不可用时显式报错，不回退内存假数据

## 3. 模块职责

### M1 职责

1. 将治理任务状态变化写入 `governance.tasks` 与 `governance.results`
2. 将人工审核回写写入 `governance.reviews`
3. 将全链路事件写入 `audit.events`

### M2 职责

1. 统一指标口径（完成率、阻塞率、人工介入率、平均置信度）
2. 聚合风险分布（confidence buckets / blocked reasons / low-confidence patterns）
3. 提供版本效果对比（baseline vs candidate）
4. 提供任务下钻数据（输入、输出、strategy、evidence、review、trace）
5. 提供新增治理包流水线聚合（stage_counts、end_to_end_success_rate、runtime_submit_success_rate）
6. 提供 LLM 交互聚合（success/failure、token 用量、延迟分位、失败原因 Top）

### M3 职责

1. 提供运行态只读接口
2. 参数校验与错误语义标准化
3. 保证前端无需自行计算核心指标
4. 对 LLM 交互默认返回脱敏摘要；admin 导出全量需审计

### M4 职责

1. 呈现运行态业务结果
2. 支持筛选联动和下钻追溯
3. 空态时给出“样例包灌入引导”

### M5 职责

1. 构造覆盖高置信/复核/阻塞的样例数据
2. 灌入后在 24h 观测窗口内可见
3. 为演示与回归测试提供可重复数据基线

## 4. 数据流

### F1. 生产运行流

1. CLI/Agent/API 提交治理任务
2. Runtime Worker 执行治理并写入结果
3. 审核员回写人工审核
4. 审计事件持续写入 `audit.events`
5. Observability Domain Service 从 PG 聚合查询
6. Dashboard UI 调用 Runtime Observability API 展示

新增治理包链路：

1. 用户/测试客户端访问 Factory CLI 发起治理包新增流程
2. Factory Agent 与 LLM 对话确认治理需求并生成工作包
3. 工作包提交到 Governance Runtime，写入 submit/accept/run/finish 事件
4. 审计事件与链路事件统一写入 `audit.events`
5. Observability API 聚合为 pipeline/llm/event timeline 数据供页面展示

### F2. 样例灌入流

1. Seed Pack Loader 生成 60+ 地址样例
2. 批量调用任务提交接口写入运行库
3. 对需复核样例回写审核决策
4. UI 刷新后显示完整运行态指标与明细

### F3. 下钻追溯流

1. 用户点击任务 ID
2. API 查询 task/result/review/audit
3. 返回 evidence 列表和时间线
4. UI 展示任务证据链与状态演进

### F4. 工作包链路追溯流

1. 用户按 `workpackage_id/version/client_type` 筛选
2. API 查询 `created -> llm_confirmed -> packaged -> submitted -> accepted -> running -> finished` 阶段事件
3. API 返回阶段漏斗、节点成功率、时延分位与 receipt
4. UI 弹窗展示 CLI/Agent/LLM/Runtime 时间线

## 5. API 边界

### 5.1 北向边界（对前端/外部）

1. 前端仅调用 `runtime/*` 观测 API
2. 不暴露数据库表结构
3. 不提供研发流程看板 API 作为本页面主数据源

### 5.2 东西向边界（服务内部）

1. Router -> Observability Service -> Repository
2. 不允许 Router 直接拼装 SQL
3. 不允许 UI 调用运行时内部 worker 方法

### 5.3 南向边界（基础设施）

1. 仅 Repository/DAO 访问 PG
2. 仅 Queue Adapter 访问队列
3. 观测服务只读运行库，不修改任务状态

## 6. 依赖关系

## 6.1 依赖方向

1. `Dashboard UI -> Observability API`
2. `Observability API -> Observability Domain Service`
3. `Observability Domain Service -> Repository`
4. `Repository -> PostgreSQL(governance/runtime/audit)`
5. `Seed Pack Loader -> Governance API`

## 6.2 技术依赖

1. FastAPI（接口层）
2. PostgreSQL（唯一业务真相源）
3. Pydantic（契约模型）
4. 前端模板/静态资源渲染组件

## 7. 不允许的耦合方式

1. Dashboard 直接读库
2. 前端本地计算核心 KPI 代替后端统一口径
3. 观测模块反向控制 Runtime 状态机
4. 用研发过程数据（Story/研发主题/CI）充当运行态业务指标
5. 数据缺失时回退 `in_memory` 或 Dummy runtime 伪造结果
6. 跳过审核记录而直接写“已复核”状态
7. 非 Alembic 路径变更生产 schema

## 8. 数据模型约束（运行态观测最小集合）

1. `governance.tasks`：任务主记录（status/ruleset/version/timestamps）
2. `governance.results`：治理输出（confidence/strategy/normalized/evidence_refs）
3. `governance.reviews`：审核记录（reviewer/outcome/reason/reviewed_at）
4. `audit.events`：时序事件（task_id/event_type/payload/created_at）
5. `runtime.workpackage`：工作包主记录（workpackage_id/version/checksum/submit_status/runtime_receipt_id）
6. `audit.events.payload` 必须可提取：`client_type/pipeline_stage/workpackage_id/version/trace_id/span_id`
7. LLM 事件必须记录：`model/base_url/latency_ms/token_usage/success/failure_reason`（默认脱敏）

## 9. 性能与容量基线（MVP）

1. 默认查询窗口 `24h`，支持 `1h/7d/30d`
2. 列表分页默认 `limit=50`
3. 首页 4 个聚合 API 总响应目标 `< 1.5s`（本地环境）
4. 明细下钻响应目标 `< 800ms`（单任务）
