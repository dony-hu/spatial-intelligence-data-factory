# UX 设计与验收：Runtime Observability V2 / S2-15

> 文档状态：当前有效（研发主题正式交互文档）
> 融合来源：页面设计稿 + 人机确认闭环交互规格 + 原型验收清单 + Factory Agent 交互补充约束

## 1. 目标

为 `OBS-RUNTIME-S2-15` 提供可实现、可测试的页面设计输入，支撑以下链路：

1. 选择 `workpackage_id@version` 执行上传批次。
2. 查看多轮确认摘要与人工确认节点。
3. 查看试运行报告（逐地址结果 + 批次图谱）。
4. 下钻事件时间线（中文可读）并关联 Runtime 回执。

## 2. 页面信息架构

1. 顶部筛选区
- 时间窗：`1h/24h/7d/30d`
- 工作包：`workpackage_id@version`
- 客户端类型：`user/test_client`

2. 上传执行区（新增 S2-15 重点）
- CSV 上传控件
- 工作包选择器（必选）
- 执行按钮
- 执行结果提示（成功/阻断）

3. 人机确认摘要区（新增 S2-15 重点）
- 对话轮次统计
- 每轮摘要（`goal/constraint/decision`）
- 确认动作时间线：`confirm_generate -> confirm_dryrun_result -> confirm_publish`

4. 试运行报告区（新增 S2-15 重点）
- 输入数据摘要（总条数、来源）
- `records[]` 结果表（标准化/拆解/验证）
- 批次级 `spatial_graph` 卡片（唯一图谱）

5. 事件下钻区
- `workpackage-events` 列表
- 中文字段展示：`source_zh/event_type_zh/status_zh/description_zh/pipeline_stage_zh`
- JSON 模式切换

## 3. 关键交互定义

1. 上传执行
- 用户上传 CSV 后，若未选择 `workpackage_id@version`，执行按钮禁用并提示。
- 点击执行后展示任务创建状态与 `runtime_receipt_id`。

2. 试运行查看
- 点击“查看试运行报告”弹窗，先展示输入摘要，再展示 `records[]` 表，再展示批次图谱。
- 当图谱状态为 `PARTIAL` 时，必须显示 `failed_row_refs`。

3. 时间线下钻
- 点击某事件行展开 `payload_summary`。
- 提供“中文叙述/JSON 原文”切换，默认中文叙述。

## 4. 状态与错误设计

1. 空态
- 无工作包时：提示先创建工作包。
- 无数据时：提示先上传 CSV 或执行种子灌入。

2. 错误态
- `400 INVALID_PAYLOAD`：参数冲突或缺失（如仅传 `workpackage_id`）。
- 门禁阻断：明确显示“未确认动作，禁止进入下一阶段”。
- 运行异常：显示 `blocked/error`，禁止展示伪成功。

3. 成功态
- 上传成功显示 `task_id`、`workpackage_id@version`、`runtime_receipt_id`。

## 7. 验收口径（UI 视角）

1. 可执行：上传 CSV + 选择工作包可触发执行。
2. 可解释：对话摘要、确认节点、中文事件描述可直接阅读。
3. 可追溯：试运行报告与 Runtime 回执可关联同一 `workpackage_id@version`。
4. 可测试：核心元素具备稳定 `data-testid`，支持 Web E2E 自动化。

## 8. 人机确认闭环补充约束

1. 人机协同链路必须完整可见：`CLI -> Agent -> LLM -> 人工确认 -> Runtime`。
2. 页面必须可按 `workpackage_id@version` 执行并回查。
3. 确认时间线固定展示：`confirm_generate -> confirm_dryrun_result -> confirm_publish`。
4. 每个确认节点必须可查看审计字段：`actor/action/decision/reason/timestamp/trace_id/workpackage_id/version`。
5. 默认“中文叙述”模式，支持切换 JSON 原文。

## 9. 原型最小验收清单

1. 门禁阻断：
   - 默认未确认发布时，点击“上传并执行”显示阻断横幅。
   - 执行前若缺少 `workpackage_id@version`，按钮禁用或提示错误。
2. 试运行报告：
   - 报告必须包含输入摘要、`records[]`、`spatial_graph`。
   - `PARTIAL` 状态必须显示 `failed_row_refs`。
3. 事件下钻：
   - 事件列表可切换“中文叙述/JSON”。
   - `source_zh/event_type_zh/status_zh/description_zh/pipeline_stage_zh` 非空。
4. E2E 可测性：
   - `wp-selector`
   - `csv-upload-input`
   - `upload-exec-button`
   - `confirm-timeline-panel`
   - `dryrun-report-open`
   - `dryrun-records-table`
   - `dryrun-graph-card`
   - `events-table`
   - `event-view-mode-toggle`
   - `runtime-receipt-id`
   - `gate-blocking-banner`

## 10. Factory Agent 交互补充约束

1. 页面必须存在对话输入与发送动作，并展示用户消息与 Agent 回复。
2. 页面必须展示 Agent -> LLM 请求与 LLM -> Agent 响应，以及模型、耗时、token 或摘要信息。
3. 页面必须展示工作包阶段流转：创建、收敛、确认、打包、试运行、发布。
4. 页面必须展示工作包产物结果：`id/version/规则摘要/发布状态`。
5. 页面必须允许录入或粘贴输入样本，并展示逐条处理过程与批次级结果摘要。
6. 治理执行前必须从下拉选择 `workpackage_id@version`，不可自由输入。
7. 工作包列表必须说明“来源于 Agent ↔ LLM 自动维护”，并能看到最近维护记录。
