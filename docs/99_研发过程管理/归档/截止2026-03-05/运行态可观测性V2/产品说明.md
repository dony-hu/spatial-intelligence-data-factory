# 产品说明：系统运行态可观测页面

> 文档属性：阶段性产品说明
> 现行正式设计以 `docs/07_系统运行与运维/系统可观测性能力设计.md`、`docs/05_数据模型设计/数据库分域设计.md`、`docs/04_系统组件设计/03_Runtime执行/Agent与Runtime交接契约.md` 为准。

## 1. 背景与问题

当前页面偏向研发过程与工程事件，不满足业务运营视角。用户需要看到“系统运行成果”，包括：

1. 实际治理任务处理量与成功率
2. 低置信度/阻塞原因分布
3. 人工审核效率与积压
4. 规则集与版本对结果质量的影响
5. 可追溯到任务、批次、样本地址的证据链
6. 可追溯到“新增治理包”的全链路过程：用户/测试客户端访问工厂 CLI、Factory Agent 与 LLM 交互、工作包提交到治理 Runtime

## 2. 产品目标

构建一个“系统运行态”可观测页面，面向运营、治理负责人、值班人员，支持以下核心场景：

1. 每日巡检：系统是否健康运行，是否可继续发布/执行
2. 异常处置：快速定位阻塞任务与主要失败原因
3. 质量运营：识别低置信地址类型，评估人工审核闭环效果
4. 版本评估：不同 ruleset/workpackage 版本的质量对比
5. 新增治理包观测：确认从 CLI 发起到 Runtime 接收执行的链路是否完整、稳定、可审计

## 3. 目标用户

1. 数据治理运营（主用户）
2. 值班工程师（次用户）
3. 业务负责人（查看周报/月报）

## 4. 页面定位与边界

### 4.1 页面定位

页面名称：`系统运行态可观测总览`

目标：展示“系统运行结果与质量”而非“研发流程状态”。

### 4.2 不在本期范围

1. 研发任务管理（story、研发主题、研发进度）
2. CI/CD 构建细节
3. 代码评审过程指标

## 5. 信息架构（页面模块）

### M1. 运行总览卡片（L0）

1. 今日任务总数
2. 今日完成率（`completed / total`）
3. 今日阻塞率（BLOCKED / total）
4. 平均置信度
5. 待审核积压数
6. 最新一次处理时间

### M2. 质量与风险（L1）

1. 置信度分层分布（`>=0.85`、`0.60~0.85`、`<0.60`）
2. Top5 阻塞原因（按 task 数）
3. Top5 低置信地址模式（按规则：缺门牌、道路歧义、POI 冲突等）
4. 审核处理时长分位（P50/P90）

### M3. 规则与版本效果（L2）

1. ruleset 维度成功率对比
2. workpackage 版本治理效果对比（平均置信度、阻塞率、人工介入率）
3. 近 7 天质量趋势（成功率、阻塞率、平均置信度）

### M4. 任务明细与追溯（L3）

1. 可筛选任务列表（状态、ruleset、时间窗、置信度区间）
2. 单任务下钻：输入地址、标准化结果、strategy、evidence、审核记录
3. trace 回放链接（事件时间线）

### M5. 新增治理包链路观测（L3）

1. 新增治理包流水线总览：`created -> llm_confirmed -> packaged -> submitted -> accepted -> running -> finished`
2. 链路时延分解：CLI 交互时延、Agent 规划时延、LLM 往返时延、Runtime 提交时延
3. 节点成功率：CLI 请求成功率、LLM 调用成功率、Runtime 提交成功率、端到端成功率
4. 交互轨迹：用户/测试客户端请求摘要、Agent 决策摘要、LLM 请求/响应摘要（脱敏）
5. 工作包提交明细：`workpackage_id/version/checksum/skills_count/artifact_count/submit_status/runtime_receipt_id`

## 6. 关键筛选维度

1. 时间窗：`1h/24h/7d/30d`
2. ruleset_id
3. workpackage_id/version
4. 执行控制态（`SUBMITTED/PLANNED/APPROVAL_PENDING/APPROVED/CHANGESET_READY/EXECUTING/EVALUATING/COMPLETED/NEEDS_HUMAN/FAILED/ROLLED_BACK`）
5. 治理结果态（`completed / needs_human / blocked / failed`）
6. 置信度区间
7. 是否需人工审核
8. pipeline_stage（链路阶段）
9. client_type（user/test_client）

## 7. 指标定义（统一口径）

1. 任务完成率 = `completed_tasks / total_tasks`
2. 阻塞率 = `BLOCKED / total_tasks`
3. 人工介入率 = `reviewed_tasks / total_tasks`
4. 平均置信度 = `avg(confidence)`（仅有结果样本）
5. 待审核积压 = `pending_review_tasks`
6. 审核时长 = `reviewed_at - task_created_at`
7. 端到端时延 = `runtime_accepted_at - cli_request_at`
8. LLM 往返时延 = `llm_response_at - llm_request_at`
9. Runtime 提交成功率 = `runtime_submit_success / runtime_submit_total`
10. 新增治理包闭环率 = `finished_workpackages / created_workpackages`

## 8. API 设计（建议）

### 8.1 运行总览

`GET /v1/governance/observability/runtime/summary?window=24h&ruleset_id=...`

返回：

1. `total_tasks`
2. `status_counts`
3. `avg_confidence`
4. `pending_review_tasks`
5. `reviewed_tasks`
6. `latest_task_at`

### 8.2 风险分布

`GET /v1/governance/observability/runtime/risk-distribution?window=24h`

返回：

1. `confidence_buckets`
2. `blocked_reason_top`
3. `low_confidence_pattern_top`

### 8.3 版本效果对比

`GET /v1/governance/observability/runtime/version-compare?baseline=...&candidate=...`

返回：

1. `success_rate_delta`
2. `blocked_rate_delta`
3. `avg_confidence_delta`

### 8.4 任务明细

`GET /v1/governance/observability/runtime/tasks?...`

返回：

1. 列表项含 `task_id/status/ruleset_id/confidence/strategy/review_status/updated_at`
2. 分页信息

### 8.5 新增治理包链路总览

`GET /v1/governance/observability/runtime/workpackage-pipeline?window=24h&client_type=...`

返回：

1. `total_workpackages`
2. `stage_counts`
3. `end_to_end_success_rate`
4. `latency_breakdown_ms_p50/p90`
5. `runtime_submit_success_rate`

### 8.6 链路事件时间线（CLI/Agent/LLM/Runtime）

`GET /v1/governance/observability/runtime/workpackage-events?workpackage_id=...&version=...`

返回：

1. `trace_id/span_id/parent_span_id`
2. `source`（`test_client|factory_cli|factory_agent|llm|governance_runtime`）
3. `event_type`
4. `occurred_at`
5. `status`
6. `payload_summary`（脱敏摘要）

### 8.7 LLM 交互观测（脱敏）

`GET /v1/governance/observability/runtime/llm-interactions?window=24h&workpackage_id=...`

返回：

1. `model/base_url/request_count/success_count/failure_count`
2. `latency_ms_p50/p90`
3. `token_usage_prompt/completion/total`
4. `failure_reasons_top`
5. `samples`（仅返回脱敏后的 prompt/response 摘要）

## 9. 页面交互草图（文本）

1. 顶部：筛选条（时间窗、ruleset、状态）
2. 第一行：6 个 KPI 卡片
3. 第二行左：质量趋势图；右：阻塞原因 Top5
4. 第三行左：置信度分层；右：规则/版本对比
5. 第四行：新增治理包链路观测面板（阶段漏斗、时延分解、节点成功率）
6. 底部：任务明细表（可点击 task_id 下钻）+ 工作包链路事件弹窗

## 10. 验收标准（MVP）

1. 页面默认展示最近 24h 系统运行数据，不依赖研发看板文件
2. 任务量 > 0 时，KPI 与趋势图可正确渲染
3. 支持按状态与 ruleset 过滤，且结果一致
4. 点击任务可看到标准化结果与 evidence
5. 无数据时显示“空态提示 + 用例灌入引导”，不是空白页
6. 至少可查看 1 条新增治理包完整链路（CLI -> Agent -> LLM -> Runtime）的事件时间线
7. 任一工作包可展示提交凭证（receipt）与提交状态，不允许只展示“任务已创建”而无 Runtime 接收证据
8. LLM 观测数据必须脱敏展示，且可按 workpackage/version 过滤

## 11. 风险与缓解

1. 风险：业务数据不足导致页面空白
   - 缓解：提供“样例数据灌入脚本 + 一键生成按钮”
2. 风险：指标口径不一致
   - 缓解：统一由 `governance_service` 输出聚合指标
3. 风险：查询压力上升
   - 缓解：分页、时间窗约束、只读 SQL 白名单
4. 风险：LLM 交互日志包含敏感信息
   - 缓解：默认脱敏、viewer 只读摘要、admin 才可导出全量并记录审计日志

## 12. 上线计划

1. S1：API 聚合与页面骨架
2. S2：风险分布与任务下钻
3. S3：版本对比与性能优化
4. S4：新增治理包链路观测（CLI/Agent/LLM/Runtime）与端到端闭环验收

## 13. 地址治理样例包（用于页面有数据可观测）

### 13.1 目标

提供一批可重复灌入的地址治理任务，覆盖成功、低置信、阻塞、人工复核闭环，让页面在 24h 窗口内稳定展示“运行成果”。

### 13.2 用例结构（建议最少 60 条）

1. A 类（高置信自动通过）20 条
2. B 类（中置信需复核）20 条
3. C 类（低置信高风险）10 条
4. D 类（阻塞/失败）10 条

### 13.3 地址样本覆盖维度

1. 城市层级：一线/新一线/县域
2. 场景类型：住宅、园区、商圈、乡镇、道路交叉口
3. 异常类型：缺门牌、别名冲突、道路重名、POI 失效、行政区变更
4. 输入质量：完整地址、半结构化地址、噪声地址

### 13.4 每条任务最小字段

1. `task_id`
2. `input_address`
3. `ruleset_id`
4. `workpackage_version`
5. `status`
6. `confidence`
7. `blocked_reason`（可空）
8. `review_required`
9. `review_outcome`（可空）
10. `evidence_refs`

### 13.5 灌入后页面预期结果（用于验收）

1. 任务总数 `>= 60`
2. 完成率在 `55%~80%`
3. 阻塞率在 `10%~25%`
4. 人工介入率在 `25%~45%`
5. 阻塞原因 Top5 与低置信模式 Top5 非空
6. 任务明细表首屏可见多状态混合数据

## 14. 页面改版原则（系统运行态，不是研发过程）

### 14.1 信息层级

1. 第一优先级：治理产出与质量
2. 第二优先级：风险与异常定位
3. 第三优先级：版本差异与策略优化

### 14.2 必须移除或弱化的信息

1. Story/研发主题/开发进度状态
2. 构建日志、测试流水线明细
3. 研发任务追踪型卡片

### 14.3 页面成功判定

当值班运营在 3 分钟内能回答以下问题，视为设计成功：

1. 当前系统是否在稳定处理治理任务？
2. 主要风险来自哪类地址/哪条规则？
3. 今天是否需要增加人工审核班次？
4. 最新版本是否比基线版本更好？

## 15. MVP 验收补充（面向业务运行）

1. 支持“无研发上下文”独立访问：仅依赖治理运行库数据
2. 首次打开页面可直接看到样例治理结果（通过样例包灌入）
3. 所有图表可点击联动到任务明细
4. 任一任务可下钻到输入、输出、evidence、审核结论
5. 页面所有指标均可追溯到同一套 API 聚合口径
6. 页面可观测“新增治理包”全过程，不依赖研发看板：用户/测试客户端 -> 工厂 CLI -> Factory Agent ↔ LLM -> Runtime 提交/执行
7. 链路观测支持按 `workpackage_id/version/client_type` 联动筛选，并可下钻到事件时间线
