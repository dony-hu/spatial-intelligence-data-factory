# Story: S1-A1 治理任务状态机稳定化

## 目标

保证 Runtime 控制态在主链路中按现行状态机合法推进，并保证对外结果摘要不会回退到旧的 `PENDING/RUNNING/SUCCEEDED` 简化口径。

## 验收标准

1. 任务提交后初始状态进入 `SUBMITTED`，规划后进入 `PLANNED`。
2. 审批路径只允许按 `APPROVAL_PENDING -> APPROVED` 或失败路径演进。
3. 执行路径只允许按 `CHANGESET_READY -> EXECUTING -> EVALUATING -> COMPLETED/FAILED/NEEDS_HUMAN/ROLLED_BACK` 演进。
4. 任一异常路径不允许返回空状态、旧状态别名或未定义状态。

## 开发任务

1. 先补测试：增加非法跳转、审批分支、失败分支和人工复核分支的状态流转失败用例。
2. 再改实现：校正 `tasks` 路由与 `governance_job` 状态写入时机，并对齐 Runtime Orchestrator 状态机。
3. 最后验证：运行目标测试集并输出状态流转证据。

## 测试用例

1. 正常提交+执行：最终进入 `COMPLETED`。
2. 模拟 worker 异常：最终进入 `FAILED`。
3. 审批或评估需要人工介入：最终进入 `NEEDS_HUMAN`。
4. 非法状态跳转：直接拒绝并留下错误证据。

## 对齐信息（PRD/架构）

1. PRD 对齐：EPIC A-A1（状态机）+ EPIC B-B1/B2（运行模式可控）。
2. 架构对齐：
- `docs/02_总体架构/系统总览.md` 治理主链路状态机要求。
- `docs/02_总体架构/模块边界.md` API/Worker/Core 边界。
- `docs/04_系统组件设计/03_Runtime执行/Runtime调度与任务系统.md` Runtime 控制态定义。

## 模块边界与 API 边界

1. 所属模块：`governance_api`、`runtime worker`、`governance repository`。
2. 上游入口：任务提交与调度触发。
3. 下游依赖：状态持久化、审计写入。
4. API 边界：状态查询从数据库读取，不读取进程内临时状态作为真相源。

## 依赖与禁止耦合

1. 允许依赖：`api/worker -> repository`。
2. 禁止耦合：
- 状态机逻辑散落在脚本侧并绕过统一服务层。
- 非定义状态写入数据库或返回体。
