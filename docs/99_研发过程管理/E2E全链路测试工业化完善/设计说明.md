# 设计说明

## 1. 设计原则

本主题遵循以下原则：

1. 正式测试文档保留稳定真相源，过程管理区负责推进整改，不混淆两者职责。
2. E2E 不再只写“场景名称”，而要写成可直接落自动化的详细规格。
3. 任何阻塞、失败、人工介入场景都必须落到状态机、证据和责任域。
4. 血缘验证必须同时覆盖正向链和逆向链，不能只做结果尾部回查。
5. 所有整改保持 docs-only，不越过代码边界。

## 2. 结构化拆解

### 2.1 正式文档补齐

补齐或增强以下正式文档：

1. `docs/09_测试与验收/E2E用例模板.md`
2. `docs/09_测试与验收/全链路测试设计.md`
3. `docs/09_测试与验收/测试方案总览.md`
4. `docs/09_测试与验收/测试用例目录.md`

### 2.2 设计口径收敛

本主题要求 E2E 设计与以下正式专题保持一致：

1. Agent 状态机
2. Runtime 状态机
3. Agent 与 Runtime 交接契约
4. 数据血缘与可追溯设计
5. 数据湖与执行技术架构

### 2.3 缺口映射

| 评审缺口 | 整改策略 |
|---|---|
| 只有目录级用例，没有详细规格 | 新增正式模板并要求必填字段 |
| 构建阻塞链路缺失 | 单独拆 Story，覆盖 key、依赖、能力缺口 |
| Runtime 审批/回滚缺失 | 单独拆 Story，要求验证状态推进和证据 |
| `blocked` 归因模糊 | 引入责任域与原因码字段 |
| 血缘只有尾部回查 | 新增全链血缘 E2E 要求 |
