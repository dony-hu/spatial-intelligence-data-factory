# 软件架构与技术选型分析报告

> 文档状态：当前有效
> 角色：Ring2 过程分析报告
> 对应正式文档：`docs/02_总体架构/软件架构设计.md`

## 1. 报告目标

本报告用于沉淀本轮关于模块编程语言、系统开源骨架和 Agent 框架的评估结论。它解决的不是“今天代码已经怎么写”，而是“后续软件架构应优先朝什么方向收敛”。

本报告的使用边界如下：

1. 它是过程分析与选型依据。
2. 它不替代 `docs/02_总体架构/` 下的正式架构真相源。
3. 稳定结论已同步回填到 `docs/02_总体架构/软件架构设计.md`。

## 2. 现状判断

### 2.1 当前实现更像“自研编排壳”，不是成熟 Agent 骨架

综合当前正式文档与实现，可得到以下判断：

1. `FactoryAgent` 已经承担了 intent routing、结构化记忆、蓝图重试、schema 校验、门禁状态与 trace 输出等编排职责。
2. `NanobotAdapter` 当前更接近模型与 Provider 访问适配层，而不是拥有完整状态机与恢复语义的主 Loop 框架。
3. `OpenCodeWorkpackageBuilder` 当前负责 bundle 文件落盘与脚本模板生成，仍然是 Builder，而不是独立 Agent runtime。
4. 文档中的 `nanobot <-> opencode` 语义已经存在，但在当前代码里仍偏向“协议语义 + 本地适配调用”，而不是强一等公民的 A2A 主循环。

### 2.2 当前技术栈的真实优势

当前实现虽然还不是最终骨架，但有三个真实优势：

1. Python 生态已经覆盖 Agent、Schema、API、脚本生成和测试主链路。
2. `workpackage_schema.v1` 与 `orchestration_context.v1` 已经形成正式契约，说明系统不是“自由聊天代理”，而是契约驱动的编排系统。
3. `WAIT_USER_INPUT / WAIT_USER_GATE / BLOCKED` 已经被设计成显式状态，而不是隐藏在 prompt 里的隐式行为。

### 2.3 当前方案的主要短板

1. 主 Loop 所有权分散在 `FactoryAgent`、`nanobot` 适配层和本地 Builder 之间。
2. 长生命周期暂停、恢复、人工介入和回放的能力目前主要靠自研状态对象维持，维护成本高。
3. Runtime 调度、服务契约、Agent 编排和观测目前缺少统一开源骨架，各层能力的工程化程度不均衡。

## 3. 模块编程语言分析

### 3.1 结论总览

| 模块 | 当前适配语言 | 推荐主语言 | 受控备选 | 结论 |
|---|---|---|---|---|
| Factory Agent 编排层 | Python | Python 3.11 | Google ADK 多语言化后再评估 Go/Java/TS | 保持 Python，优先引入成熟 Agent 编排框架 |
| 北向 API（治理 / 观测 / 管理台后端） | Python | Python 3.11 | Go 仅在高并发内部服务时引入 | 当前 repo、Pydantic、LLM 工具链与 Python 更贴合 |
| Runtime Orchestrator / Worker 控制层 | Python | Go | Argo Workflows 作为 K8s-first 备选 | 这是最值得跳出当前实现重新选型的部分 |
| Workpackage Executor | Python + Shell | Go 控制容器 + Python 3 / Shell 入口 | Rust 隔离执行器 | 控制层与执行脚本层分离，不把服务语言和工作包语言混为一体 |
| Bundle 内数据处理脚本 | Python + Shell | Python 3 + Shell | SQL / R / Java / Scala 受控扩展 | 与当前正式工作包入口契约一致 |
| 前端工作台 / 管理台 | TypeScript | TypeScript | 无 | 前端不应继续塞入 Python 服务中混做 |
| 高隔离执行器 / 沙箱 Runner | 无统一基线 | Rust | Go | 只在隔离、安全和资源密度要求明显上升时引入 |

### 3.2 为什么 Factory Agent 继续优先用 Python

1. 当前主链路已经围绕 Python 形成：
   - LLM 调用与工具适配
   - JSON Schema 校验
   - workpackage 生成
   - 测试与 CLI
2. 适合本项目的 Agent 框架当前也主要集中在 Python 生态。
3. 编排层关注的是状态、结构化输出、人机协作和工具调用，而不是极限 CPU 性能。

### 3.3 为什么 Runtime 调度层建议转向 Go

1. Runtime 的核心难点是长生命周期调度、状态推进、重试、租约、超时、并发和故障恢复。
2. 这类问题更接近“服务编排与分布式控制面”，而不是“LLM 编排与结构化输出”。
3. Go 在并发、部署、可运维性、团队维护成本和与工作流引擎集成方面更均衡。

### 3.4 Rust 的适用边界

Rust 不应一上来就作为全系统主语言，但适合两类场景：

1. 执行器需要更强隔离时。
2. bundle 运行密度和安全边界要求显著提高时。

因此 Rust 更适合作为后续执行器增强位，而不是当前一阶段的软件主骨架。

## 4. 开源骨架分析

### 4.1 Runtime 骨架

| 方案 | 定位 | 优点 | 风险 | 建议 |
|---|---|---|---|---|
| [Temporal](https://docs.temporal.io/) + Go SDK | 长生命周期工作流与任务执行骨架 | 适合显式状态机、重试、暂停/恢复、人工介入、持久化 workflow history | 引入专用工作流基础设施，需要新的运行与监控习惯 | 推荐主基线 |
| [Argo Workflows](https://argo-workflows.readthedocs.io/en/stable/) + [Argo Events](https://argoproj.github.io/argo-events/) | Kubernetes-first 工作流与事件骨架 | 适合容器化任务编排、artifact 管理、集群内调度 | 偏 K8s-first；对非容器化 bundle 和服务编排不如 Temporal 自然 | 作为 K8s-first 备选 |
| Redis / RQ（现状） | 轻量队列与分发 | 现有代码已接入，改造成本最低 | 不是完整工作流骨架，暂停/恢复/人工门禁需要自研 | 保留为过渡态，不作为长期主骨架 |

结论：

1. 如果目标是平台级 Runtime，优先 `Temporal + Go`。
2. 如果组织明确是全量 `Kubernetes-first`，再考虑 `Argo Workflows + Argo Events`。
3. 不建议继续把 `Redis / RQ` 当长期 Runtime 骨架。

### 4.2 服务契约与接口骨架

| 方案 | 定位 | 建议 |
|---|---|---|
| [Buf](https://buf.build/docs/generate/) + Protobuf | 多语言契约管理、lint、breaking-check、代码生成 | 推荐作为内部服务契约基线 |
| [ConnectRPC](https://connectrpc.com/docs/introduction/) | 兼容浏览器、gRPC、gRPC-Web 的服务接口层 | 推荐作为内部服务 RPC 备选；北向 REST/JSON 仍可保留 |
| FastAPI + Pydantic | 北向 HTTP / JSON API | 继续作为当前阶段北向 API 的主骨架 |

结论：

1. `workpackage_schema` 继续使用 JSON Schema。
2. 服务间结构化契约新增 `Buf + Protobuf`。
3. 北向 API 不强制一次性切到 gRPC，优先保留 `FastAPI` 的开发效率和现有兼容性。

### 4.3 可观测骨架

| 方案 | 定位 | 建议 |
|---|---|---|
| [OpenTelemetry](https://opentelemetry.io/docs/) | 统一埋点、trace、metric、log 语义与上下文传播 | 推荐作为统一埋点骨架 |
| [Prometheus](https://prometheus.io/docs/introduction/overview/) | 指标采集、存储与告警 | 推荐作为指标与告警骨架 |
| 自有 DB + Dashboard 闭环 | 任务状态、证据、回放、审计查询 | 必须保留，不被外部平台替代 |

结论：

1. OpenTelemetry 负责统一埋点和 trace 上下文。
2. Prometheus 负责指标与告警。
3. PostgreSQL + 自有观测域仍然是正式查询与回放真相源。

### 4.4 事件总线骨架

| 方案 | 定位 | 建议 |
|---|---|---|
| [NATS JetStream](https://docs.nats.io/nats-concepts/jetstream) | 高效事件流、持久消费者、异步通知 | 作为可选事件总线，不作为业务真相源 |
| 直接 DB 轮询 / 纯队列 | 简单但耦合高 | 不建议继续扩大使用范围 |

结论：

1. `JetStream` 适合事件通知、回放触发、异步风控等辅助场景。
2. 不应用 `JetStream` 替代 PostgreSQL 的结构化真相源，也不把它当对象产物层。

## 5. Agent 框架分析

### 5.1 评估标准

本次对 Agent 框架的判断依据不是“谁最流行”，而是是否满足以下要求：

1. 支持长生命周期、有状态主 Loop。
2. 支持显式人机介入和恢复。
3. 支持结构化输出与契约校验。
4. 能和现有 `orchestration_context.v1`、`workpackage_schema.v1` 对齐。
5. 能和后续 Runtime、观测、测试门禁协同。

### 5.2 方案对比

| 方案 | 优点 | 风险 | 建议 |
|---|---|---|---|
| 当前自研方案：`FactoryAgent + nanobot + opencode` | 改造最小；贴现有测试与 schema | 主 Loop 所有权分裂；暂停/恢复/HITL 维护成本高 | 保留为过渡态与兼容方案 |
| [LangGraph](https://docs.langchain.com/oss/python/langgraph/overview) + [PydanticAI](https://ai.pydantic.dev/) | LangGraph 适合 long-running stateful agents；PydanticAI 强于 typed output、approval、durable hooks | 需要把现有状态机与记忆对象正式映射成 graph state | 推荐主方案 |
| [Google ADK](https://google.github.io/adk-docs/) | Python / TS / Go / Java 多语言统一潜力强；区分 LLM Agent、Workflow Agent、Custom Agent | 迁移成本高；和当前 Python/Pydantic 流程衔接需要额外设计 | 作为战略备选 |
| [OpenAI Agents SDK](https://developers.openai.com/api/docs/guides/agents-sdk) | handoff、session、guardrails、tracing 完整 | 供应商绑定偏强，不适合当系统根骨架 | 适合做专用子 Agent，不适合做主框架 |

### 5.3 当前方案与推荐方案的关键差异

| 对比点 | 当前方案 | 推荐方案 |
|---|---|---|
| 主 Loop 所有权 | 分散在 `FactoryAgent`、`nanobot` 适配和本地 Builder | 统一由 LangGraph 状态机承载 |
| 结构化输出 | 自研逻辑 + JSON Schema 校验 | PydanticAI typed output + JSON Schema / validator 双重校验 |
| 人工介入 | 自研 `WAIT_USER_INPUT / WAIT_USER_GATE` | LangGraph interrupt + approval，状态仍映射现有语义 |
| 记忆模型 | 自研 `orchestration_context.v1` 对象 | 保留 `orchestration_context.v1`，作为 graph state contract |
| nanobot 角色 | 混合了“推理入口”和“主 Loop 语义” | 降级为模型与 Provider 访问适配层 |
| opencode 角色 | Builder + 构建协议语义 | 继续做 Builder / 工具节点，不再承担主 Loop 语义 |

## 6. 推荐组合

### 6.1 推荐主组合

推荐的主组合如下：

1. Agent 编排：
   - `LangGraph + PydanticAI + nanobot 适配层`
2. Runtime 调度：
   - `Temporal + Go`
3. 北向 API：
   - `FastAPI + Pydantic`
4. 服务契约：
   - `Buf + Protobuf`
5. 内部 RPC：
   - `ConnectRPC` 作为内部服务通信优选
6. 可观测：
   - `OpenTelemetry + Prometheus + PG 观测域`
7. 事件总线：
   - `NATS JetStream` 作为可选辅助总线
8. Bundle 运行入口：
   - `Python 3 + Shell`

### 6.2 保留和降级策略

1. 保留 `workpackage_schema.v1` 与 `orchestration_context.v1`。
2. 保留 `opencode` 作为工作包 Builder / 工具节点。
3. 保留 `nanobot` 作为模型和 Provider 适配层。
4. 降级当前自研 `FactoryAgent` 的主 Loop 角色，逐步把状态机迁移到正式 Agent 骨架中。

## 7. 分阶段建议

### 7.1 第一阶段：文档与契约收敛

1. 先把模块语言、Agent 框架和 Runtime 骨架写入正式软件架构设计。
2. 保持 `workpackage_schema.v1` 与 `orchestration_context.v1` 不变。
3. 给现有 `FactoryAgent` 标出“编排壳 / 兼容层 / 过渡层”定位。

### 7.2 第二阶段：Agent 主 Loop 收敛

1. 用 LangGraph 映射：
   - `DISCOVERY`
   - `ALIGN_IO`
   - `BLUEPRINT_LOOP`
   - `BUILD_WITH_OPENCODE`
   - `VERIFY`
   - `PUBLISH`
2. 用 graph state 承接：
   - `interaction_state`
   - `blocker_ticket`
   - `gate_state`
   - `timeline`
3. nanobot 只保留为模型访问节点。

### 7.3 第三阶段：Runtime 骨架升级

1. 引入 Temporal 与 Go runtime control plane。
2. 保持 Worker 仍按 `workpackage_id@version` 调起 bundle 入口。
3. 明确：
   - Temporal workflow state 不替代 PostgreSQL 业务真相源
   - bundle 语言不因 Runtime 服务语言切换而变化

### 7.4 第四阶段：内部契约和观测升级

1. Buf + Protobuf 固化内部服务契约。
2. OpenTelemetry 统一 trace 和 metrics 埋点。
3. Prometheus 承接告警与 SLI / SLO。

## 8. 不建议作为主骨架的方向

本轮不建议作为主骨架的方向包括：

1. 继续把 `Redis / RQ` 扩展成长期 Runtime 主骨架。
2. 继续把 `nanobot` 直接当作系统级主 Loop，而不补正式状态机框架。
3. 直接把全系统重写为 Rust。
4. 直接把 `OpenAI Agents SDK` 作为全系统根骨架。
5. 把 `NATS JetStream` 或工作流引擎自身状态，当作业务正式真相源。

## 9. 对正式文档的回填结论

本次分析稳定沉淀为以下 Ring1 结论：

1. Factory Agent 主语言继续以 Python 为主。
2. Runtime 控制层优先采用 Go + 正式工作流骨架。
3. Bundle 入口继续保持 Python 3 / Shell。
4. Agent 正式框架优先采用 LangGraph + PydanticAI。
5. Runtime 正式骨架优先采用 Temporal。
6. 服务契约优先采用 Buf + Protobuf。
7. 可观测统一骨架优先采用 OpenTelemetry + Prometheus。
