# BMAD 文档设计说明

> 文档状态：当前有效
> 角色：BMAD 与当前 `docs/` 主线、SpecKit 规格主线的协同设计

## 1. 设计原则

1. `docs/` 编号目录和 `docs/99_研发过程管理/` 是正式真相源。
2. `_bmad-output/` 是工作流过程产物，不是最终文档主线。
3. 同一类知识只允许一个正式归口位置，避免 BMAD 与 SpecKit 双写。
4. 规划中的文档可以先由 BMAD 定题，但只有在内容稳定后才进入编号目录。

## 4. BMAD 到 SpecKit 的交接接口

当出现以下条件时，应从 BMAD 主题切到 SpecKit feature：

1. 主题目标和 Story 边界已经稳定。
2. 需要细化接口契约、数据模型、任务拆解。
3. 需要在 `specs/<feature-id>/` 中维护 `spec.md / plan.md / tasks.md / research.md / data-model.md / contracts/`。

交接时必须携带以下最小上下文：

1. `theme_id`
2. `story_id`
3. 关联真相源路径
4. 验收路径
5. 受影响的正式文档目录

## 5. SpecKit 回填 BMAD 的规则

SpecKit 不直接替代 BMAD 主题文档。模块规格稳定后，按以下方式回填：

1. Story 范围与 DoD 变化：回写 `docs/99_研发过程管理/<主题>/主题说明.md` 或 `故事/*.md`
2. 冲刺节奏变化：回写 `docs/99_研发过程管理/冲刺计划.md`
3. 项目状态变化：回写 `docs/99_研发过程管理/工作流状态.yaml`
4. 长期架构决策：回写 `docs/02_总体架构/` 或相关编号目录

## 6. 初始化后的使用方式

1. 先从 `docs/00_阅读指南.md` 和 `docs/02_总体架构/架构索引.md` 建立背景。
2. 用 BMAD 生成或维护项目级主题、Story、冲刺文档。
3. 对高风险模块或复杂接口，再切到 SpecKit 做 feature 级规格。
4. 在 `文档映射配置.yaml` 中维持路径级映射，不再依赖口头约定。
