# 研发过程管理目录说明

> 文档状态：当前有效
> 角色：`docs/99_研发过程管理/` 使用说明
> 生效日期：2026-03-06

## 1. 目的

`docs/99_研发过程管理/` 是合并后的唯一过程管理目录，用于统一承接：

1. 项目级流程文件
2. 研发主题文档
3. 历史过程归档

## 2. 目录约定

当前目录分三层：

1. 根目录：项目级过程文件，例如 `工作流状态.yaml`、`冲刺计划.md`、`规格融合/`
2. 主题目录：当前有效的研发主题
3. `归档/`：历史过程文档

每个研发主题目录至少包含：

1. `主题说明.md`
2. `故事/`

按需要补充：

1. `产品说明.md`
2. `设计说明.md`
3. `交互设计.md`
4. `验收/`
5. `测试/`
6. `评审记录.md`
7. `复盘记录.md`
8. `冲刺计划.md`

## 3. 规则

1. 研发主题目录内的正式文件名不带日期后缀。
2. 阶段性 PRD、UX、验收应融合为研发主题目录下的正式文件，而不是持续新增 dated 文件。
3. `2026-03-05` 及之前的历史过程文档统一迁入 `docs/99_研发过程管理/归档/截止2026-03-05/`。
4. 新增过程主题时，若它服务于正式文档整改或架构收敛，应在目录说明里登记主题入口，避免隐性过程线继续扩散。

## 4. 使用边界

1. `docs/99_研发过程管理/` 是过程管理区，不是正式阅读主线。
2. 稳定产品、架构、测试、运维文档优先归入编号目录。
3. 某个主题阶段完成后，真正沉淀为长期真相源的内容应再归口到编号目录。

## 5. 对 AI Coding 的生效方式

`docs/99_研发过程管理/文档分层索引.yaml` 不是普通说明文档，而是后续 AI Coding 的机器可读约束源。

后续 Codex、BMAD、SpecKit 或其他仓库内 AI 协作流程，应按下面顺序执行：

1. 先读取 `AGENTS.md` 与 `docs/99_研发过程管理/文档分层索引.yaml`
2. 先判断任务属于正式知识更新、过程推进，还是证据归档
3. 再决定写入编号目录、`docs/99_研发过程管理/`，还是 `Ring3` 归档目录
4. 若本次任务改变了正式边界，必须同步更新正式文档入口和相关索引

也就是说，`文档分层索引.yaml` 的作用不是“事后说明文档放哪”，而是“事前裁决 AI 任务该读什么、能写什么、完成后还要补什么”。
