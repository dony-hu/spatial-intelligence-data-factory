# Retrospective：EPIC16 Runtime 控制层替换试点（2026-03-09）

## 1. 保留项（Keep）

1. 先冻结 `Go Runtime / Go Executor / truth-source` 合同信号，再推进默认主链切换。
2. 始终守住 `workpackage_id@version` 唯一执行面和 PostgreSQL 正式真相源边界。
3. 用最小 backend smoke 和合同回归共同证明“替换成立”，而不是只凭实现说明收口。
4. 每个 Story 收口时同步回填 Ring1 正式文档和 Ring2 过程状态，避免“代码已完成、正式文档仍停留在旧基线”。

## 2. 删除项（Drop）

1. 让 `Redis / RQ` 继续以“当前正式主链”身份留在总体架构与部署文档中的做法。
2. 在 Ring1 正式文档里固化某一次 `passed` 数量、warning 数和具体测试报告路径的做法。
3. 通过 legacy entrypoint fallback 掩盖 bundle 未产出 `runtime_output.json` 的真实失败通路。

## 3. 改进项（Improve）

1. 后续任何替换型 Epic 在标记 `done` 前，都应追加一次 Ring1 / Ring2 / Ring3 文档结构审视，而不是只看代码和测试。
2. Ring1 测试文档只保留“回归入口、通过信号、truth-source 检查点”，具体运行结果统一下沉到主题验收和 Ring3 证据。
3. Runtime 相关顶层图、技术上下文和部署文档必须共用同一套“当前正式基线 / 历史兼容背景 / 受控备选”表达方式。

## 4. 下一轮硬约束（Hard Constraints）

1. Runtime workflow/control engine 不能取代 PostgreSQL 作为正式结构化真相源。
2. bundle 若缺少 `output/runtime_output.json`，必须真实阻塞，不允许 fallback 伪成功。
3. Agent、Runtime、API、Observability 对外合同变更时，必须同时更新总体架构、组件契约和测试设计入口。
4. `done` 判定必须同时满足：story 状态、主题状态、Ring1 正式文档、验收结论一致。

## 5. 责任人

1. A-ARC：维护 Runtime 当前正式基线与总体架构口径一致。
2. A-DEV：维护 `Go Runtime + Go Executor` 默认主链与 no-fallback 约束。
3. A-QA：维护合同回归入口、backend smoke 和 truth-source 检查点。
4. A-SM：维护 Epic 收口时的状态同步、复盘和过程文档清尾。
