# 设计说明

## 1. 设计原则

本主题遵循以下原则：

1. 外壳不变，内部换核。
2. 先迁移主 Loop，再迁移复杂支线。
3. 先迁移 Agent 编排层，不提前替换 Runtime 控制层。
4. `nanobot` 保留为 adapter，不再承担系统级主框架职责。
5. 采用单向迁移，不保留旧编排链路切换机制。
6. 任何状态迁移都必须保持正式结构化契约与测试口径可追踪。

## 2. 目标结构

### 2.1 Agent 编排层新角色分工

本主题的目标角色分工如下：

1. `LangGraph`
   - 承担 Agent 状态机与主 Loop。
2. `PydanticAI`
   - 承担类型化输出、工具调用、approval / interrupt 语义。
3. `nanobot`
   - 保留为真实外部 LLM 调用与 Provider 访问适配层。
4. `opencode`
   - 保留为 Builder / 工具节点，不承担主 Loop。
5. `FactoryAgent`
   - 保留 façade，对外维持当前稳定合同。

### 2.2 正式状态集合映射

本主题默认以当前正式状态集合为唯一试点范围：

1. `DISCOVERY`
2. `ALIGN_IO`
3. `BLUEPRINT_LOOP`
4. `BUILD_WITH_OPENCODE`
5. `VERIFY`
6. `PUBLISH`
7. `WAIT_USER_INPUT`
8. `WAIT_USER_GATE`
9. `BLOCKED`
10. `COMPLETED`

要求如下：

1. 新 graph state 必须能表达这些状态。
2. `WAIT_USER_INPUT / WAIT_USER_GATE` 必须映射到正式 interrupt / approval 语义。
3. `BLOCKED` 必须保持显式原因与恢复点，不回退到自然语言等待。

## 3. 结构化拆解

### 3.1 状态图与 graph state

首批迁移重点是把当前编排记忆和状态推进映射成正式 graph state：

1. `orchestration_context.v1` 作为正式结构化输入。
2. graph state 消费并回写：
   - `boot_context`
   - `discovery_facts`
   - `capability_snapshot`
   - `blueprint_attempts`
   - `opencode_task_ticket`
   - `gate_state`
   - `runtime_evidence`
   - `publish_decision`
   - `timeline`

### 3.2 迁移切片

为控制试点风险，本主题按两批切片迁移：

1. 第一批：
   - `DISCOVERY`
   - `ALIGN_IO`
   - `BLUEPRINT_LOOP`
2. 第二批：
   - `BUILD_WITH_OPENCODE`
   - `VERIFY`
   - `PUBLISH`
   - 人机中断与阻塞恢复分支

### 3.3 façade 保持稳定

本主题不直接移除 `FactoryAgent`，而是要求：

1. `FactoryAgent` 继续作为 façade。
2. CLI、API、测试优先调用 façade，不直接穿透到新 graph 内核。
3. 替换动作默认发生在 façade 背后，并直接切换到新编排内核。
4. 本主题不保留旧编排链路的长期并行接线。

## 4. 测试与验收策略

本主题的测试策略如下：

1. 不再要求新旧 Agent 链路长期并行对照。
2. 合同回归测试应覆盖：
   - requirement query
   - general governance chat
   - blueprint generation
   - dryrun / publish 触发
   - human gate / blocked / recover
3. `orchestration_context.v1` 映射必须进入结构化合同测试。
4. 对外 façade 合同测试必须持续保护 CLI / API 行为不破。

## 5. 与后续主题的关系

本主题完成后，才适合启动：

1. Runtime 控制层替换试点
2. 内部服务契约升级试点
3. 更大范围的 Agent / Runtime 协同替换
