# 研发主题：Agent框架替换试点

## 1. 主题目标

在 EPIC4 已完成重构实施基线落位的前提下，正式启动 Agent 编排层的框架替换试点，解决以下问题：

1. 当前 `FactoryAgent + nanobot + opencode` 仍然更接近自研编排壳，主 Loop、状态机、人机中断和恢复语义分散在多处实现中。
2. 正式软件架构已经明确 Agent 编排层应向 `LangGraph + PydanticAI + nanobot adapter` 收敛，需要一个可控试点把架构结论转成实现基线。
3. 后续 Runtime 控制层替换、内部服务契约升级和更大范围的多 Lane 实施，需要先完成一次受控范围内的单向迁移，验证新的 Agent 骨架能稳定承接当前正式状态集合和人机协同语义。

## 2. In Scope

1. 在不破坏当前 CLI / API / publish / dryrun 外部合同的前提下，启动 Factory Agent 主 Loop 的单向替换试点。
2. 将当前正式状态集合映射到 `LangGraph` 状态图，包括：
   - `DISCOVERY`
   - `ALIGN_IO`
   - `BLUEPRINT_LOOP`
   - `BUILD_WITH_OPENCODE`
   - `VERIFY`
   - `PUBLISH`
   - `WAIT_USER_INPUT`
   - `WAIT_USER_GATE`
   - `BLOCKED`
   - `COMPLETED`
3. 将 `orchestration_context.v1` 映射为 graph state 的正式结构化对象。
4. 用 `PydanticAI` 承担类型化输出、工具调用和 approval / interrupt 语义。
5. 保留 `nanobot` 作为模型 / Provider 访问适配层。
6. 保留 `opencode` 作为 Builder / 工具节点。
7. 建立新编排内核的合同回归测试与替换验收口径。

## 3. Out of Scope

1. 不在本主题内正式切换 Runtime 到 `Go + Temporal`。
2. 不在本主题内全量落地 `Buf + Protobuf + ConnectRPC`。
3. 不在本主题内重写 `governance_worker` 或 bundle 执行链。
4. 不在本主题内进行前端大规模重构。
5. 不在本主题内替换 `nanobot` 为新的 Provider 访问层。
6. 不把本主题误做成“全系统最终重构完成”。

## 4. Story Backlog

1. `EPIC15-S1`：Agent 状态机与 graph state 映射设计固化
2. `EPIC15-S2`：FactoryAgent façade 保持稳定并切换到新编排内核
3. `EPIC15-S3`：`DISCOVERY / ALIGN_IO / BLUEPRINT_LOOP` 试点迁移
4. `EPIC15-S4`：`WAIT_USER_INPUT / WAIT_USER_GATE / BLOCKED` 语义迁移
5. `EPIC15-S5`：新编排内核合同回归测试与替换验收

## 5. 完成定义（DoD）

满足以下全部条件视为本主题完成：

1. `LangGraph` 状态图已能承接首批正式状态集合。
2. `orchestration_context.v1` 已形成新的 graph state 映射口径。
3. `PydanticAI` 已接管类型化输出、工具调用和 approval / interrupt 语义。
4. `nanobot` 已明确降级为模型 / Provider 访问适配层，不再承担 Agent 主框架职责。
5. 对外 CLI / API / publish / dryrun 合同保持稳定。
6. 新编排内核已形成合同回归测试与替换验收结论。
