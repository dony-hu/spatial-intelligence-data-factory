# 共享契约冻结清单

## 1. 目的

本清单用于把 Epic13 在并行开发启动前必须冻结的共享契约对象显式列出，并为每个对象指定 owner、变更路径和下游消费方。

使用原则：

1. 本清单只回答“哪些对象不能自由并发修改，以及谁说了算”。
2. 本清单优先服从 `AGENTS.md`、`docs/99_研发过程管理/文档分层索引.yaml` 与 `docs/10_研发与工程规范/多Agent与多Worktree并行开发规范.md`。
3. 下游 Lane 在共享契约未合并前，不得提前实现猜测版接口或结构。

## 3. 变更路径规则

### 3.1 只允许 owner 串行修改

以下对象不允许多个 Lane 自由并发编辑：

1. `AGENTS.md`
2. `docs/00_阅读指南.md`
3. `docs/02_总体架构/架构索引.md`
4. `docs/10_研发与工程规范/`
5. `docs/99_研发过程管理/文档分层索引.yaml`
6. `migrations/versions/`

处理方式：

1. 由唯一 owner 在专属分支上修改。
2. 修改前需说明影响的下游 Lane。
3. 合并前由集成值班确认没有并发冲突窗口。

### 3.2 必须先契约 PR，后下游实现

以下对象允许演进，但必须先冻结共享口径：

1. `workpackage_schema/schemas/`
2. `contracts/`
3. Agent 与 Runtime 交接契约
4. 跨域数据库 / API 契约
5. `services/governance_api/app/models/` 中影响多 Lane 的共享字段
6. 正式架构文档中影响多个 Lane 的接口、边界、依赖关系

处理方式：

1. 先提交独立契约 PR。
2. PR 里必须说明：
   - owned surface
   - 影响对象
   - 下游消费方
   - 最小回归要求
3. 契约 PR 合并后，下游 Lane 才能开始实现 PR。

## 4. 契约 PR 与实现 PR 拆分规则

### 4.1 必须单独提契约 PR 的情况

1. 修改共享 schema、共享 API 载荷、共享状态字典。
2. 修改跨域数据库边界、迁移 owner、跨界访问口径。
3. 修改 Ring0 / Ring1 中会影响多个 Lane 的正式边界描述。
4. 修改红区对象且下游会据此实现代码或测试。

### 4.2 可以在 Lane 内实现 PR 中完成的情况

1. 不影响共享契约的局部实现细节。
2. 仅限本 Lane owned surface 的私有代码重构。
3. 仅限本 Lane 的局部测试补充或门禁修正。
4. 不会引发其他 Lane 重新解释接口的文档补充。

## 5. 集成值班门禁

1. 每日只允许进入一个共享契约类 PR。
2. 集成值班位负责确认“今天合哪一个契约 PR”。
3. 下游 Lane 若未收到契约合并确认，不得开始消费该对象。
4. 合并前至少确认：
   - owner 审核通过
   - 受影响下游已知晓
   - 本对象对应的最小测试 / 检查单已通过

## 6. 下游消费说明

1. `lane-03` 完成本清单维护后，可直接进入共享 schema / contracts 冻结。
2. `lane-04` 以本清单中的迁移 owner 和数据契约 owner 为基线。
3. `lane-06` 以本清单中的共享对象矩阵建立回归范围和通知对象。
4. `PAR-S4`、`PAR-S5` 应直接引用本清单，不再重新发明一版 owner 口径。
