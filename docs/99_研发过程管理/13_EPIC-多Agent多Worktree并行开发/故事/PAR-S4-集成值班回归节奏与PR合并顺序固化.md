# Story 13.4（PAR-S4）：集成值班、回归节奏与 PR 合并顺序固化

状态：done

说明：本 Story 对应 Epic13 的第 4 个实施故事，BMAD 数字化标识按 `13-4` 使用，当前 canonical 文件仍保留 `PAR-S4` 文件名。

## 用户故事

作为并行开发阶段的规划 / 集成编排 owner，
我希望在共享契约冻结之后，明确集成值班职责、每日回归节奏和 PR 合并顺序，
以便各 Lane 知道什么时候可以提 PR、谁负责看守 `main`、回归失败后如何停线和恢复，而不是把所有冲突堆到最后统一爆炸。

## 验收标准

1. 形成一份当前有效的集成值班与合并顺序规则，至少覆盖：
   - 集成值班 owner 的职责边界
   - 每日共享契约 PR 的进入规则
   - Lane 自测、交叉回归、合并窗口的顺序
   - 回归失败时的停线与恢复路径
2. 明确每日回归节奏，至少能回答：
   - 何时确认“今天唯一的共享契约 PR”
   - 何时完成 Lane 内最小测试集
   - 何时进入跨 Lane 回归窗口
   - 何时允许合并到 `main`
3. 明确首轮并行开发的推荐合并顺序，至少覆盖：
   - 共享契约 PR 优先
   - `lane-01`、`lane-02`、`lane-04` 的最小切片合并顺序
   - `lane-05` 仅在接口稳定后进入
   - `lane-06` 负责跨 Lane 回归 gate
4. 明确失败回退规则，至少说明：
   - 哪些情况必须暂停合并
   - 谁负责下发回归失败结论
   - 失败修复应回到哪个 Lane / Worktree 处理
   - 不允许以“先合再补救”绕过门禁
5. 本 Story 的输出仅修改过程 / 规范文档，不把集成治理逻辑硬编码进运行时代码链路。
6. 交付结果能够被后续 `PAR-S5`、`lane-06` 和各 Lane PR 模板直接消费。
