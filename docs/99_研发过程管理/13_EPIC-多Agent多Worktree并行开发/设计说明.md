# 设计说明

## 1. 设计原则

本主题遵循五条原则：

1. 先冻结共享契约，再让多个 Agent 并行实现。
2. 一条工作流只对应一个 Agent、一个 Worktree、一个分支。
3. 正式文档、数据库边界和 Schema 是上游约束，不是收尾补丁。
4. 主链集成必须有专门值班角色，不能依赖“大家自觉”。
5. Worktree 解决的是隔离问题，不是治理问题；治理问题仍由 Ring0 / Ring1 文档裁决。

## 2. 本主题的结构化拆解

### 2.1 Lane 切分

建议把后续重构切为六条 Lane：

1. 工厂 Agent 编排
2. Runtime 执行
3. Workpackage Schema 与生成契约
4. Trust Hub 与数据契约
5. Frontend 与交互
6. QA / E2E / 集成门禁

### 2.2 Worktree 隔离

每条 Lane 各自使用独立 Worktree，并隔离：

1. Python 虚拟环境
2. 本地环境变量
3. 端口
4. 数据库名 / Schema 名前缀
5. 运行产物目录

### 2.3 红区与串行修改

以下对象不能放任并发修改：

1. `AGENTS.md`
2. `docs/00_阅读指南.md`
3. `docs/02_总体架构/`
4. `workpackage_schema/schemas/`
5. `migrations/versions/`

对这些对象，只允许：

1. 指定唯一 owner
2. 先发契约 PR，再下游消费

### 2.4 集成值班

需要一个单独的集成值班角色，负责：

1. 看守 `main`
2. 处理跨 Lane 冲突
3. 控制 PR 合并顺序
4. 触发日常回归
5. 回填正式文档入口

### 2.5 过程映射

并行开发时，以下对象必须一一对应：

1. 一个 Linear Issue / Story
2. 一个 branch
3. 一个 Worktree
4. 一个 PR
5. 一个 owner

### 2.6 重构启动策略

针对“全面重构即将启动”的场景，本主题补充以下启动策略：

1. 不采用 Big Bang 一次性重写。
2. 先做 `48h contract sprint`，先把共享契约和启动顺序固定，再允许各 Lane 扩散实现。
3. 先稳边界，再迁目录；目录迁移本身不应先于契约冻结。
4. 旧实现可以过渡保留，但不允许继续在旧壳上扩张新能力。

### 2.7 `48h contract sprint` 的目标

在首个 `48h contract sprint` 内，需要先产出以下结果：

1. 冻结 `workpackage_schema.v1` 与 `orchestration_context.v1` 的共享契约口径。
2. 冻结 Agent 与 Runtime 最小交接载荷及状态映射口径。
3. 冻结数据库分域、迁移 owner 和红区 owner。
4. 固化首轮 Lane 启动顺序、PR 顺序和每日集成时刻。
5. 建立首版最小回归基线与已知失败清单。

### 2.8 Lane 启动顺序

首轮重构不要求 6 条 Lane 同时进入业务实现，而是按以下顺序启动：

1. 规划 / 集成编排位先启动，负责冻结 owner、合并顺序和跨 Lane 风险。
2. `lane-03 Schema / Contracts` 先启动，负责输出共享契约 PR。
3. `lane-01 Factory Agent`、`lane-02 Runtime / API`、`lane-04 Trust Hub / Data` 在共享契约合并后并行进入实现。
4. `lane-05 Frontend` 在 API 与数据契约冻结后再接入。
5. `lane-06 QA / 集成门禁` 先建立 baseline，再在各 Lane 首轮实现后执行跨 Lane 回归。

### 2.9 集成节奏

并行开发阶段的集成节奏补充如下：

1. 每日只允许进入一个跨 Lane 共享契约类 PR，避免红区冲突叠加。
2. 集成值班位负责确认“今天合哪一个 PR”，而不是让多个 Lane 自由抢占主干。
3. 每个 Lane 合并前必须声明受影响契约、下游通知对象和最小测试集。
4. 若共享契约未合并，下游 Lane 不得自行推导接口或提前落地实现。

### 2.10 过渡态代码治理

当前仓库处于“新旧实现并存”的过渡态，因此补充以下治理要求：

1. `src/` 视为 legacy/compat 过渡面，默认不再承接新增正式能力。
2. 新能力默认落入目标 owned surface，不以“先在旧目录补一下”为理由继续扩张旧壳。
3. 若测试仍绑定旧执行链路，应优先修正测试口径，使其对齐当前正式边界。
4. 目录迁移、测试迁移与契约迁移必须拆开执行，避免一条 PR 同时改三层基线。

### 2.11 单人启动模式

针对当前“先把 epic 启动起来，再进入真实并行开发”的阶段，补充以下执行约束：

1. 本轮只完成启动物料，不模拟多个 Agent 已在不同 Lane 同时进入业务实现。
2. 当前 Codex 仅承担“规划 / 集成编排位”的启动 owner，负责核对 Worktree、补齐过程文档和输出启动检查清单。
3. 单人启动阶段不得一边声明“并行开发已启动”，一边在多个 Worktree 之间交替推进业务代码实现。
4. 无法在本地闭环的外部动作，例如 Linear 建单，必须显式记录为 blocker，而不是以占位文本冒充完成。
5. 单人启动完成后，后续是否进入真实多 Agent / 多 Lane 开发，仍以共享契约冻结和 Linear 映射就绪为前置条件。

## 3. 产出清单

1. 正式规范文档
2. 研发主题与 Story backlog
3. 项目级约束回填
4. 阅读入口与目录说明更新
5. `48h contract sprint` 冻结清单与启动顺序
6. 集成值班节奏与首轮 PR 顺序
7. 单人启动检查清单与 blocker 说明
