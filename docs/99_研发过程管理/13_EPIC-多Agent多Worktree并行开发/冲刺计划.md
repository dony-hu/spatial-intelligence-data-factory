# 冲刺计划

## 1. 目标

本轮冲刺不修改业务代码，目标是把“多 Agent / 多 Worktree 并行开发”从规范推进到可执行状态。

本轮采用单人启动模式，由当前 Codex 兼任“规划 / 集成编排位”启动 owner，负责把主题文档、Worktree 基线、门禁口径和启动检查清单补齐；不在本轮伪装成多个 Agent 已同时进入实现。

本轮完成口径：

1. 建立产品战略规划位与 6 条执行 Lane 的独立 Worktree。
2. 为每条 Lane 固化 branch、目录、owned surface 和最小门禁。
3. 明确共享契约冻结顺序、集成值班职责和合并顺序。
4. 为后续 Linear 建单和 PR 派工提供一一映射基线。

## 4. 串行冻结顺序

并行开发前，按下面顺序冻结共享对象：

1. `lane-03` 先冻结 `workpackage_schema/`、`schemas/`、`contracts/` 的共享契约。
2. 规划 / 集成编排位确认红区 owner、PR 顺序和 Story 映射。
3. `lane-01`、`lane-02`、`lane-04` 在共享契约冻结后并行实现。
4. `lane-05` 在 API / 数据契约稳定后接入。
5. `lane-06` 最后补齐跨 Lane 回归矩阵并执行门禁。

3. Ring0 文档仅由规划 / 集成编排位统一回填，不由执行 Lane 分散修改。

## 6. 首个 `48h contract sprint` 启动节奏

### 6.1 第 0-24 小时

1. 规划 / 集成编排位确认红区 owner、合并窗口和每日集成时刻。
2. `lane-03` 输出共享契约冻结清单，至少覆盖：
   - `workpackage_schema.v1`
   - `orchestration_context.v1`
   - Agent 与 Runtime 交接载荷
   - 数据库分域与迁移 owner
3. `lane-06` 输出首版 baseline：
   - 最小 smoke 集
   - 已知失败清单
   - 环境依赖与验证入口

### 6.2 第 24-48 小时

1. 规划 / 集成编排位合入首个共享契约 PR。
2. `lane-01`、`lane-02`、`lane-04` 基于已合并契约开始实现各自最小切片。
3. `lane-05` 仅消费已冻结接口，不自行推导 API。
4. `lane-06` 对首轮实现产物执行最小跨 Lane 回归。

### 6.3 启动期附加规则

1. 每日只进入一个跨 Lane 共享契约类 PR。
2. 共享契约未合并前，下游 Lane 不得提前实现猜测版接口。
3. 单个 PR 不同时承接“契约变更 + 目录迁移 + 大规模实现迁移”三类改动。
4. 若发现测试仍绑定旧链路，优先修正测试 seam，再继续功能改造。

## 7. Linear 与 PR 映射要求

正式进入开发前，每条 Lane 必须补齐：

1. 一个 Linear Issue 或 Story。
2. 一个 branch。
3. 一个 Worktree。
4. 一个 PR。
5. 一个 owner。

PR 描述至少包含：

1. owned surface
2. 影响的共享契约
3. 最小测试集
4. 需要通知的下游 Lane

## 8. 收尾后外部后续

1. 为 6 条执行 Lane 分别在 Linear 中补建 Story，并绑定对应 owner；若本地无有效鉴权，则保留为人工后续项。
2. 各 Lane 在自己的 Worktree 内补齐本地 `.venv`、`.env.local`、端口和数据库隔离策略，再进入真实业务实现。
3. 业务实现开始前，再做一次红区、依赖关系和测试口径复核。
4. 正式进入多 Lane 业务开发时，继续复用本主题已经冻结的契约、值班、映射和 gate 规则。
