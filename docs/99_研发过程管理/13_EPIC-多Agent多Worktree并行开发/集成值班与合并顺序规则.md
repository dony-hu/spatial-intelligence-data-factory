# 集成值班与合并顺序规则

## 1. 目的

本规则用于把 Epic13 并行开发进入真实实现前的集成值班职责、每日节奏、合并顺序和失败恢复路径固化下来。

它与 `共享契约冻结清单.md` 的关系是：

1. `共享契约冻结清单.md` 负责回答“哪些对象不能自由并发修改、谁说了算”。
2. 本规则负责回答“谁在什么时刻看守 `main`、什么 PR 先合、失败后回哪条 Lane 修”。

## 2. 角色与职责

| 角色 | 主职责 | 不负责的事项 |
| --- | --- | --- |
| 规划 / 集成值班位 | 确认当日唯一共享契约 PR、安排回归窗口、决定是否允许合并到 `main`、下发停线结论 | 不替代各 Lane 修业务代码 |
| Lane owner | 在自己的 Worktree 内补齐最小测试集、准备 PR 描述、修复回归失败 | 不得跳过值班位直接推进到 `main` |
| 契约 owner | 对共享契约类改动给出最终 owner 审核意见 | 不负责下游 Lane 的实现细节 |
| `lane-06` QA / 集成门禁 | 执行最小 smoke 与跨 Lane 回归、输出 gate 结论、维护已知失败清单 | 不承担业务实现 owner 职责 |
| PR 作者 | 提供 owned surface、影响契约、最小测试集、下游通知对象 | 不得省略门禁字段或伪造结果 |

## 3. 启动期每日节奏

### 3.1 建议时刻基线（Asia/Shanghai）

| 阶段 | 建议时刻 | owner | 输入 | 输出 |
| --- | --- | --- | --- | --- |
| 当日共享契约确认 | `11:00` 前 | 集成值班位 | 候选契约 PR、受影响 Lane 列表 | “今天唯一共享契约 PR”结论或“今日无契约 PR” |
| Lane 自测收口 | `14:00` 前 | 各 Lane owner | Lane 内最小测试、检查清单 | 可进入交叉回归的 PR 候选 |
| 跨 Lane 回归窗口 | `16:00` 前后 | `lane-06` + 集成值班位 | 候选 PR、已知失败清单、验证入口 | `PASS / BLOCKED / NEEDS-FIX` 结论 |
| 合并窗口 | `18:00` 前后 | 集成值班位 | 回归结论、owner 审核结论 | 允许合并 / 暂停合并 |

说明：

1. 上述时刻是当前启动期建议基线，不是长期不可变规则。
2. 若当天没有共享契约 PR，可直接从 Lane 自测收口进入交叉回归窗口。
3. 若任一前置阶段未完成，后续阶段自动顺延，不允许假装通过。

### 3.2 每日节奏强约束

1. 每日只允许一个共享契约类 PR 进入合并窗口。
2. 共享契约 PR 未合并前，下游 Lane 不得开始消费其结果。
3. Lane 自测未完成，不得进入 `lane-06` 跨 Lane 回归。
4. `lane-06` 未给出明确 gate 结论，不得进入合并窗口。

## 4. 推荐合并顺序

### 4.1 宏观顺序

1. 共享契约 PR 优先。
2. 契约合并后，再进入共享实现面较重的后端 / 数据 Lane。
3. 接口稳定后，前端 Lane 才能进入。
4. `lane-06` 不承担业务功能合并，而是给出最终 gate 结论。

### 4.2 首轮启动期推荐顺序

1. `lane-03` 或规划 / 集成编排位的共享契约 PR。
2. `lane-02 Runtime / API` 的最小实现切片。
3. `lane-04 Trust Hub / Data` 的最小实现切片。
4. `lane-01 工厂 Agent` 的最小实现切片。
5. `lane-05 Frontend` 的消费型切片。
6. `lane-06` 发布跨 Lane gate 结果并决定是否允许进入 `main`。

补充规则：

1. 若某个 `lane-04` 变更涉及 `migrations/versions/`，仍按唯一 owner 串行处理，不与其他红区改动并发。
2. 若某个 `lane-02` 变更触及 `services/governance_api/app/models/` 共享字段，必须先满足契约 owner 审核。
3. 若任一 Lane 发现仍需补契约，必须退出当前实现队列，回到共享契约 PR 阶段。

## 5. 合并前门禁

每个准备进入 `main` 的 PR 至少满足：

1. PR 描述包含：
   - owned surface
   - 影响的共享契约
   - 最小测试集
   - 需要通知的下游 Lane
2. Lane owner 已完成本 Lane 最小测试或人工检查清单。
3. 若触及共享契约或红区对象，相关 owner 已完成审核。
4. `lane-06` 已完成最小 smoke / 交叉回归并给出结论。
5. 集成值班位已确认当前窗口没有更高优先级的共享契约 PR 冲突。

## 6. 失败停线与恢复规则

### 6.1 必须暂停合并的情况

1. `lane-06` 回归结果为 `BLOCKED` 或 `NEEDS-FIX`。
2. 共享契约 owner 或集成值班位指出当前 PR 违反红区规则。
3. 同一天出现第二个共享契约 PR 想进入合并窗口。
4. 发现下游 Lane 正在消费未合并的猜测版接口。

### 6.2 恢复路径

1. 由集成值班位下发暂停合并结论，并说明 blocker。
2. 问题回到原始 Lane / Worktree 修复，不允许在 `main` 直接热修主线实现。
3. 修复后重新跑本 Lane 最小测试，再回到 `lane-06` gate。
4. 重新获得 `PASS` 前，不得恢复合并队列。

### 6.3 已合并后发现问题的处理

1. 若问题出现在合并后才暴露，使用新的修复 PR 或 revert PR 处理。
2. 回滚应使用显式 `git revert` 路径，不使用破坏性历史改写。
3. 回滚后仍需重新经过 `lane-06` 最小回归，确认 `main` 恢复可用。

## 7. 对 `lane-06` 的直接输入要求

`lane-06` 在进入 gate 前，至少应收到：

1. 最小 smoke 集入口。
2. 当前已知失败清单。
3. 统一验证入口或命令。
4. 本次 PR 影响的共享契约列表。
5. 需要重点回归的下游 Lane 列表。

若上述任一输入缺失，`lane-06` 可直接判定为 `BLOCKED`，并要求 PR 作者补齐。

## 8. 下游消费说明

1. `PAR-S5` 应直接复用本规则中的 PR 描述字段、gate 结论和角色分工。
2. 各 Lane 的 PR 描述模板应直接复用本规则中的最小必填字段。
3. `lane-06` 的 baseline、已知失败清单和统一验证入口应与本规则的每日节奏直接对接。
