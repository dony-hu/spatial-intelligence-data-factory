# Epic - 地址治理 MVP 闭环（2026-02-27）

## 1. Epic 目标

先跑通一个“地址治理 MVP”最小闭环，覆盖：

1. 工厂 CLI 与工厂 Agent 的对话交互。
2. 工厂 Agent 与 LLM 的交互（用于方案确认与生成）。
3. 对话确认数据治理方案需求。
4. 治理流程试运行（dry run）。
5. 工作包发布到数据治理 Runtime（含代码、skills、配置与观测数据）。
6. 构建地址治理流水线（从输入到治理产物）。
7. 构建可信数据 Hub（沉淀外部工具/API 与可信互联网数据能力）。

## 2. In Scope

1. CLI 侧：
- 支持对话式需求确认与工作包生成入口。
- 支持最小可运行的“试运行 / 发布”命令流。

2. Agent 侧：
- 支持需求解析、治理方案确认、工作包生成。
- 支持通过 LLM（OpenCode 适配）返回可消费的结构化结果。

3. Runtime 侧：
- 支持接收并识别工作包目录结构。
- 支持执行入口脚本与观测产物写入。

4. Trust Hub 侧：
- 支持数据源注册、API 能力沉淀、可信数据存储与查询。

## 3. Out of Scope

1. 生产级高可用发布编排。
2. 全量地区规则一次性接入。
3. 复杂权限体系与多租户控制。

## 4. MVP 完成定义（DoD）

满足以下全部条件视为 MVP 跑通：

1. 通过 CLI 发起对话可完成一次地址治理需求确认。
2. Agent 必须通过真实 LLM 返回治理方案摘要与工作包生成决策。
3. 可完成一次治理流程试运行并产出结果与观测数据。
4. 工作包可发布到 Runtime 并可执行入口脚本。
5. Trust Hub 中可查询到至少 2 类外部能力（API/工具）及样例可信数据。
6. 全链路有可追踪证据（输出 JSON/Markdown）。
7. 全链路禁止 fallback；任一关键环节阻塞时必须 fail-fast 并提交人工确认（由你确认方案）后才能继续。

## 5. Epic 拆解（Story）

1. MVP-A1：CLI-Agent-LLM 对话确认治理需求。
2. MVP-A2：治理流程试运行与结果校验。
3. MVP-A3：工作包发布到数据治理 Runtime。
4. MVP-A4：地址治理流水线最小可执行构建。
5. MVP-A5：可信数据 Hub 能力沉淀与可信数据积累。
6. MVP-A6：数据库模型与持久化闭环补齐（治理域 + Runtime 发布域 + Trust Hub 贯通）。

## 6. 关键风险与缓解

1. 风险：LLM 不稳定导致结构化输出波动。  
缓解：禁止 fallback，增加严格 schema 校验；校验失败直接阻塞并提交人工确认。

2. 风险：工作包结构与 Runtime 预期不一致。  
缓解：固化工作包契约（目录、入口、skills、元数据）。

3. 风险：Trust Hub 仅存 key，缺少能力与数据闭环。  
缓解：补充能力注册与可信样例数据持久化结构。

4. 风险：LLM 运行时适配参数与 CLI 版本不一致，导致隐式绕过。  
缓解：对齐 `opencode` 调用参数，增加“真实调用/错误/阻塞”显式状态落库与 No-Fallback 门禁检查。

## 7. 数据库补齐范围（纳入本 Epic）

1. 治理主链路读写一致性：
- 修复“写库 + 读内存”断层，任务/结果/评审查询以 DB 为准。

2. Runtime 发布与工作包注册：
- 为工作包发布新增持久化实体（发布记录、版本、状态、证据引用）。
- 将 skill 注册从“仅 SQL 文件”升级为实际读写链路。

3. CLI/Agent/LLM 交互审计：
- 增加对话会话、方案确认、LLM 调用状态（success/error/blocked）记录。
- 增加阻塞原因、人工确认人、确认结论与时间戳审计字段。

4. Trust Hub 贯通：
- 打通 `packages/trust_hub` 与 `trust_meta/trust_db`，避免两套存储割裂。

5. Schema 一致性治理：
- 收敛 `trust_meta_schema.sql` 与 Alembic 迁移定义，避免双轨不一致。
