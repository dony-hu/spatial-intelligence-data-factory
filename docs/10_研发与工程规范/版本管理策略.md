# 版本管理策略

> 文档状态：当前有效
> 角色：版本、发布与变更治理正式规则
> 关联文档：
> - `docs/10_研发与工程规范/代码规范.md`
> - `docs/10_研发与工程规范/多Agent与多Worktree并行开发规范.md`
> - `docs/07_系统运行与运维/系统部署架构.md`

## 1. 版本对象

系统里至少有四类版本对象：

1. 仓库代码版本
2. 数据库迁移版本
3. 工作包版本 `workpackage_id@version`
4. 正式文档版本（通过 Git 历史体现）

## 2. 基本策略

1. 正式文档原地更新，不新增日期后缀。
2. 数据库结构变更只能通过受控迁移进入主链路。
3. 工作包版本必须显式版本化，不得覆盖式替换。
4. 发布说明和回滚说明必须可追踪。

## 3. 数据库迁移策略

1. Alembic 是数据库结构变更的正式入口。
2. 运行时 bootstrap 仅允许用于受控辅助表和兼容修复，不得替代主迁移机制扩散。
3. 新增正式物理表时，必须明确归属域、变更责任方和回滚策略。

## 4. 工作包版本策略

1. 发布以 `workpackage_id + version` 为唯一识别。
2. 同一版本不得在原路径上静默改内容。
3. dryrun 与 publish 应能回溯到同一 bundle 与证据链。

## 5. 变更记录

1. 结构性文档变更必须更新索引或入口。
2. 重要架构变更必须在相关专题或附录中留下迁移说明。
3. 回滚、补丁和人工绕行必须留下审计记录。

## 6. 分支策略

1. `main` 是唯一集成主干。
2. 进入大规模重构前，应先打正式快照 tag，并切出保留分支。
3. 并行开发分支统一使用 `codex/` 前缀。
4. 一个分支只承接一个研发主题、一个 Lane 或一组强相关变更。

## 7. Worktree 策略

1. 多 Agent 并行开发时，优先使用独立 Git Worktree，而不是共用一个工作区。
2. Worktree 目录应位于主仓外部同级目录，不得嵌套到主仓内部。
3. 每个 Worktree 默认独立维护本地环境与运行产物。
4. Worktree 的正式协作规则以《多Agent与多Worktree并行开发规范》为准。

## 8. 并行集成策略

1. 共享契约先合并，再让下游 Lane 并行实现。
2. 红区文件默认串行修改，不允许自由并发改写。
3. 合并回 `main` 前，必须先同步主干并完成最小回归。
4. 未通过集成回归的分支不得为了赶进度直接硬合并。
