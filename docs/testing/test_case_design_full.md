# 地址治理系统测试用例设计文档（完整版）

更新时间：2026-03-01 10:23:41 +0800

## 1. 文档目标
- 建立地址治理 MVP 到运行态可观测链路的统一测试设计基线。
- 明确功能、接口、数据、可靠性与安全边界的测试口径。
- 为 Epic 与 Story 验收提供可执行、可追踪、可复现的测试依据。

## 2. 测试范围
- 工厂 CLI 与工厂 Agent 交互。
- 工厂 Agent 与 LLM 交互与门禁。
- 工作包发布到治理 Runtime 的全链路。
- 地址批量治理任务执行（输入批次→治理输出→过程日志）。
- 运行态可观测页面与运行态 API。
- Trust Data Hub 与外部可信数据接入能力。

## 3. 非范围（当前阶段）
- 多租户 RBAC 全量能力（仅覆盖单租户最小权限口径）。
- 生产级压测与容量上限标定（当前仅做基础性能门禁）。
- 历史版本迁移兼容（按当前策略可清库重建）。

## 4. 测试分层策略
- 单元测试：核心算法、规则、评分、适配器、状态机。
- 组件/模块集成测试：API-Repository-DB、Worker-Queue-Runtime、Trust Hub。
- 端到端测试：CLI/Agent/Runtime/Observability 全链路行为。
- Web E2E 测试：运行态观测页面上传、任务执行、结果展示交互。
- 契约测试：关键 API 字段、状态流转、错误码、审计事件结构。

## 5. 质量门禁（No-Fallback）
- 默认禁止 in_memory fallback 通过主链路验收。
- 无真实 Runtime/LLM 依赖时，标记 blocked，不以 Dummy runtime 代替。
- PG-Only：主链路禁止 SQLite 路径；相关 SQL/DDL 需通过命名与域边界守卫测试。
- 可观测性验收必须包含：源数据、治理成果、过程日志三视图一致性。

## 6. 测试环境与数据要求
- 数据库：PostgreSQL（本地或 CI 专用实例）。
- 服务：governance_api、governance_worker、factory agent/cli。
- LLM：真实配置可用（不可用即阻塞，不降级）。
- 数据：覆盖标准地址、歧义地址、缺失字段、脏数据、多语种样本。

## 8. 入口与退出准则
### 8.1 入口
- PRD、架构、Story 验收口径已冻结到当前迭代。
- 核心依赖（PG、Worker、Runtime、LLM）可用或已明确阻塞项。
- 测试数据集与脚本可复用。

### 8.2 退出
- P0/P1 用例全部通过；P2 仅允许有明确豁免记录。
- 无 fallback 口径违规项。
- 运行态可观测页面可展示任务批次输入/输出/日志。
- 缺陷闭环：阻塞项有结论与责任人，非阻塞项进入下一迭代。

## 11. 风险与补强建议（测试经理）
- 优先保证运行态 API 与 Web 页面的一致性契约，避免‘页面有但数据语义不闭环’。
- 持续压缩 mock/stub 在主验收路径中的占比，保留在隔离测试层。
- 对‘上传批次→自动建任务→执行→可视化结果’增加稳定性回归集。
- 建立失败分层归因标签：环境阻塞/数据问题/代码缺陷/依赖波动。
